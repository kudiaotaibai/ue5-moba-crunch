# UE 渲染插件一帧是怎么跑起来的 - 小白版

## 1. 这篇文档要解决什么问题

第一篇里我们已经把这些词分别讲过了：

- Plugin
- Module
- Scene View Extension
- Global Shader
- RDG

但你现在大概率还有一个问题：

**这些东西在 UE 里到底是怎么串起来工作的？**

这篇就是专门回答这个问题的。

它不追求特别严谨的引擎源码级细节，先给你建立一条“能理解工程运行过程”的主线。

## 2. 先看最短版本

先记住这一句：

**UE 开始渲染一帧画面时，你的插件通过 Scene View Extension 拿到一个插入机会，在这个机会里用 RDG 添加自己的 Pass，Pass 里跑 Global Shader，最后把结果输出到屏幕或中间纹理。**

如果你现在只能记住一句话，就记住这一句。

## 3. 先看完整链路

把整个流程压成一条线，大概是这样：

1. UE 启动项目
2. 你的 Plugin 被加载
3. Plugin 里的 Module 启动
4. Module 注册一个 Scene View Extension
5. UE 每次开始渲染画面时，都会经过这个扩展点
6. 你的扩展点代码告诉 UE：“这里我要加一个自定义渲染步骤”
7. 这个自定义渲染步骤通过 RDG 组织成一个或多个 Pass
8. 每个 Pass 里会调用你的 Global Shader
9. Shader 在 GPU 上真正计算效果
10. 结果被写到 Render Target
11. 最终结果进入 UE 的后续流程，最后显示到屏幕

你可以把它理解成：

**插件负责把你放进车间，扩展点负责给你插队口，RDG 负责排工序，Shader 负责真正干活。**

## 4. 第 0 步：UE 启动项目

当你打开项目时，UE 不只是加载地图和角色，它还会：

- 读取项目配置
- 读取插件配置
- 加载模块
- 初始化渲染系统

这个时候，和你的渲染项目有关的第一件事就是：

**UE 会发现你的插件存在。**

如果你的插件是 Project Plugin，并且配置正确，UE 就会把它一起加载进来。

## 5. 第 1 步：Plugin 被加载

插件本身不是“会跑的代码”，它更像一个说明书和容器。

通常插件里会有：

- `.uplugin`
- `Source/你的模块`
- `Shaders/`

当 UE 读取 `.uplugin` 后，就知道：

- 这个插件叫什么
- 它有哪些模块
- 什么时候加载

所以你可以理解为：

**Plugin 负责告诉 UE：这里有一组额外功能，请把它接进项目。**

## 6. 第 2 步：Module 启动

插件真正“活过来”，靠的是模块。

当模块启动时，通常会执行类似：

- `StartupModule()`
- `ShutdownModule()`

这里就是你做初始化的地方。

对我们的项目来说，模块启动阶段最重要的工作通常是：

- 注册 Shader 目录
- 创建或注册 Scene View Extension
- 做一些全局初始化

所以你可以把模块启动理解成：

**“插件的主程序开始上班了。”**

## 7. 第 3 步：注册 Scene View Extension

这是整条链里最关键的接入点之一。

你前面已经知道：

**Scene View Extension = UE 渲染流程里的官方扩展入口**

那“注册”它是什么意思？

意思就是：

**告诉 UE：以后每当你渲染视图时，把我这个扩展对象也叫上。**

这很像你在系统里挂了一个回调：

- UE 正常渲染
- 渲染到某个阶段
- 顺手调用你注册的扩展代码

如果没有这一步：

- 你的插件虽然加载了
- 你的 Shader 虽然存在
- 但 UE 根本不知道什么时候该执行你的效果

所以这一层的作用非常明确：

**让你的代码“获得参与每帧渲染的资格”。**

## 8. 第 4 步：UE 开始渲染一帧

现在把视角切到“每一帧”。

假设此时玩家正看着场景，UE 每一帧都会大致做这些事：

- 更新游戏逻辑
- 更新相机
- 收集这一帧要画的内容
- 准备视图数据
- 开始组织渲染任务

对你的项目最重要的是：

**UE 会为当前视角准备一份 View 信息。**

里面会包含很多和这帧画面有关的数据，例如：

- 视口大小
- 相机相关矩阵
- 当前 SceneColor / SceneDepth 等上下文信息

你可以理解成：

**UE 正准备开始画这一帧，并把“这帧需要哪些信息”整理好了。**

## 9. 第 5 步：渲染来到你的扩展点

当 UE 走到 Scene View Extension 对应的阶段时，它会调用你扩展对象里的某个函数。

小白阶段你不用死记每个具体函数名，先理解这件事：

**你的扩展对象会在每帧渲染中获得一次“插入自定义渲染逻辑”的机会。**

这时候你通常已经能接触到这些东西：

- 当前 View
- 当前帧的渲染上下文
- RDG Builder
- 可能的输入纹理或输出目标

也就是说，从这一步开始：

**你终于拿到了真正动手加效果的位置。**

## 10. 第 6 步：在这个入口里使用 RDG

现在轮到 RDG 出场。

你已经知道 RDG 的本质是：

**帮你组织资源和 Pass 依赖关系。**

所以在扩展点函数里，你不会直接说：

- “显卡，马上执行这个 Shader！”

而更像是在告诉 RDG：

- 我需要一个中间纹理
- 我需要一个 Pass
- 这个 Pass 要读取哪些输入
- 这个 Pass 要写到哪里

也就是：

**你不是直接蛮干，而是在给 RDG 提交一份渲染施工计划。**

## 11. 第 7 步：创建 RDG 资源

很多效果都要中间结果，所以第一件常见的事是创建资源。

比如：

- 一张 AO 结果图
- 一张 Blur 后的结果图
- 一张 Debug 输出图

这些资源通常不是长期常驻的，而是这帧里临时用到的。

这也是 RDG 的价值之一：

**你只声明“我这帧需要这些资源”，RDG 会帮你管理它们的生命周期。**

你可以把它理解成：

**像工厂临时领用工具，用完就归还，不需要你手动一直背着。**

## 12. 第 8 步：准备 Shader 参数

Shader 不是凭空运行的，它需要参数。

例如一个全屏效果常见会需要：

- 输入纹理
- 采样器
- 屏幕大小
- 强度参数
- 半径参数

这些参数一般由 C++ 这边准备，然后传给 Shader。

所以这一步的本质是：

**CPU 侧把“这次计算需要什么输入”整理好，交给 GPU。**

这也是你以后会经常看到各种 `Parameters` 结构体的原因。

## 13. 第 9 步：添加 RDG Pass

资源和参数准备好之后，就轮到真正添加 Pass。

这一步可以理解成：

**向 RDG 说：请帮我安排一项具体渲染任务。**

一个 Pass 会说明：

- 用哪个 Shader
- 读什么
- 写什么
- 用什么参数

比如：

- Pass A：根据深度和法线计算 SSAO 原图
- Pass B：对 SSAO 原图做 Blur
- Pass C：把结果输出到屏幕或调试视图

注意：

你不是手写“先 A 再 B 再 C”的底层调度细节，  
而是把依赖关系交给 RDG，让它去整理。

## 14. 第 10 步：Global Shader 在 GPU 上真正运行

这一步是全链路里“真正出效果”的地方。

前面那些步骤都更像准备工作：

- 挂入口
- 注册回调
- 建资源
- 组参数
- 安排 Pass

而真正做计算的是：

**Shader 在 GPU 上执行。**

例如一个全屏像素着色器可能会：

- 读取当前像素周围的深度
- 计算遮蔽程度
- 输出一个灰度 AO 值

或者 Blur Shader 会：

- 读取周围多个像素
- 做加权平均
- 输出更平滑的结果

所以一定要分清：

- C++ 负责“组织”
- Shader 负责“计算”

## 15. 第 11 步：结果写回 Render Target

Shader 运行完以后，结果不会自动“神奇地出现在屏幕上”。

它必须写到某个目标里。

这个目标可能是：

- 中间纹理
- 最终输出纹理
- 某个 Debug Target

所以你可以理解为：

**每个 Pass 的结果都要落到一张图上。**

之后后续 Pass 再去读这张图，或者最终把它送到屏幕显示。

## 16. 第 12 步：UE 继续后面的渲染流程

你的效果并不会独占整条渲染链。

通常情况是：

- UE 有自己的渲染流程
- 你的效果只是其中一个插入步骤

所以你做完自己的 Pass 后，UE 还可能继续：

- 做其他后处理
- 做 UI 合成
- 做最终输出

最后画面才真正显示到显示器上。

这也解释了为什么我们总说：

**你是在“接入 UE 的渲染流程”，不是“替代 UE 的渲染流程”。**

## 17. 用一次具体例子再串一遍

假设我们以后做一个“深度可视化”效果，这条链可以这样理解：

1. 项目启动，插件加载
2. 模块启动，注册 Scene View Extension
3. UE 开始渲染某一帧
4. 轮到扩展点时，调用你的代码
5. 你的代码告诉 RDG：“我要加一个全屏 Pass”
6. 这个 Pass 读取 SceneDepth
7. 你把深度纹理和参数传给 Global Shader
8. Shader 在 GPU 上把深度转换成灰度颜色
9. 结果写到输出纹理
10. UE 再把这张结果图显示出来

如果你能顺着这 10 步在脑子里走一遍，说明你已经开始真正理解“这条链是怎么跑”的了。

## 18. 这一帧里 C++ 到底干了什么

你可以把 C++ 的职责总结成四件事：

### 18.1 进入系统

- 通过 Plugin 和 Module 挂进 UE

### 18.2 找到入口

- 通过 Scene View Extension 接入每帧渲染

### 18.3 组织流程

- 用 RDG 创建资源、组织 Pass、安排依赖

### 18.4 提供参数

- 把 View、纹理、常量参数传给 Shader

也就是说，C++ 更像“导演”和“调度员”。

## 19. 这一帧里 Shader 到底干了什么

Shader 的职责更单纯：

- 读取输入
- 做数学计算
- 输出结果

所以 Shader 更像“真正下场干活的工人”。

## 20. 最容易混淆的几个点

### 20.1 Plugin 和 Shader 不是一回事

Plugin 是容器和工程组织。  
Shader 是 GPU 代码。

### 20.2 Scene View Extension 和 RDG 不是一回事

Scene View Extension 解决的是：

- 你什么时候能插进渲染流程

RDG 解决的是：

- 插进去之后你怎么组织自己的渲染任务

### 20.3 RDG 和 Shader 不是一回事

RDG 是调度和资源组织系统。  
Shader 是真正执行图形计算的程序。

### 20.4 C++ 不负责真正画像素

C++ 负责准备和组织。  
真正处理像素、纹理、采样的是 Shader。

## 21. 你现在应该建立的核心画面感

你以后每次看到这些词，都可以在脑子里自动翻译成：

- Plugin：项目里的独立功能包
- Module：插件真正启动的代码模块
- Scene View Extension：进入每帧渲染的门
- RDG：渲染任务调度系统
- Pass：一项具体渲染步骤
- Global Shader：GPU 上执行的效果代码

然后把它们串成一句：

**模块启动后注册扩展，扩展在每帧渲染时被调用，调用里用 RDG 添加 Pass，Pass 里跑 Global Shader，结果写入纹理并最终显示。**

## 22. 这篇看完后，你至少应该能回答的问题

看完这篇后，你应该尽量能回答：

- 为什么插件加载后还需要模块启动？
- 为什么一定要有 Scene View Extension？
- 为什么不直接在普通游戏逻辑里调用 Shader？
- 为什么要先创建资源、再加 Pass？
- 为什么说 RDG 是组织系统，不是算法？
- 为什么真正的图形学计算在 Shader 里？

## 23. 最后一句话总结

这个项目里，一帧效果跑起来的核心过程就是：

**插件先接入 UE，扩展点在每帧渲染时给你一个入口，你在入口里用 RDG 安排 Pass，Pass 调用 Global Shader 在 GPU 上计算，结果写入输出纹理并进入最终显示。**
