# 纯色全屏输出要做哪几件事 - 小白版

## 1. 这篇文档解决什么问题

前面你已经知道：

- 我们第一个最适合做的效果是纯色全屏输出
- 它的目标不是好看，而是验证整条渲染链

那接下来最现实的问题就是：

**如果我要做“纯色全屏输出”，代码层面到底要做哪几件事？**

这篇不直接给你一大坨实现代码，  
而是先把任务拆清楚。

核心目标是让你明白：

**这个最小效果不是“写一个 Shader 就完了”，而是要把几个环节一起接起来。**

## 2. 先说最终目标

这个最小效果的最终目标非常简单：

**运行项目后，你能在屏幕上看到一层你指定的固定颜色。**

比如：

- 全屏偏红
- 全屏偏绿
- 全屏偏蓝

这就够了。

注意，这一步不是在追求“完整替换整个屏幕最终画面”的美术效果，  
而是在追求：

**只要你能明确看见自己的渲染结果，就说明链路已经开始打通。**

## 3. 做这个效果本质上要完成哪 5 件事

如果压缩成最核心的几件事，它其实只有这 5 步：

1. **让插件能被 UE 加载**
2. **让模块启动时完成渲染相关初始化**
3. **让每帧渲染时有一个入口能插入你的逻辑**
4. **在这个入口里添加一个全屏 RDG Pass**
5. **让这个 Pass 调用一个输出固定颜色的 Shader**

你后面看到的所有代码，其实都是围绕这 5 件事展开。

## 4. 第 1 件事：让插件能被 UE 加载

### 4.1 这里主要是谁负责

主要是：

- `.uplugin`
- `Build.cs`

### 4.2 它在纯色效果里的意义

这一步和“红色、绿色、蓝色”本身没关系，  
它只是给后面一切创造前提。

如果插件加载都不成功：

- 模块不会启动
- Shader 目录不会注册
- View Extension 不会被创建
- 你的纯色效果根本没机会参与渲染

### 4.3 这一层你要有的判断

如果后面完全没效果，  
首先要怀疑的不是 Shader 算法，而是：

- 插件有没有正确被识别和编译

## 5. 第 2 件事：模块启动时做初始化

### 5.1 这里主要是谁负责

主要是：

- `Module.cpp`

### 5.2 模块在这个效果里要干什么

对这个最小效果来说，模块主要做三件事：

1. 启动时打印日志
2. 注册 Shader 目录
3. 创建并持有 `Scene View Extension`

### 5.3 为什么打印日志也很重要

因为对小白来说，最重要的不是“代码看起来很多”，  
而是你每一步都能证明：

- 这一步确实执行了

如果模块启动时有日志，你就能先确认：

- 插件不是死的
- 模块确实起来了

这会大大减少排查焦虑。

### 5.4 为什么要先注册 Shader 目录

因为后面的 Shader 文件不是自动就能被 UE 找到。

这一步的意义是：

**告诉 UE：我插件里的 Shader 文件放在这里。**

没有这一步，后面 Shader 就算写了，也可能找不到。

### 5.5 为什么模块要创建 `Scene View Extension`

因为模块是启动入口。  
它最适合负责创建这种“长期挂在渲染流程里的全局对象”。

如果没有扩展对象：

- 你的代码没有每帧进入渲染流程的入口

## 6. 第 3 件事：每帧渲染时拿到插入机会

### 6.1 这里主要是谁负责

主要是：

- `SceneViewExtension.cpp`

### 6.2 这一步到底做什么

你不用把它想得太玄乎。

本质上就是：

**当 UE 正在渲染某一帧时，让你的代码在合适时机被叫到。**

这时你的代码终于可以说：

- 这一帧我要加一个自己的渲染步骤

### 6.3 为什么纯色效果也必须经过这里

因为你要做的是“渲染插件效果”，  
不是普通 Gameplay 逻辑。

既然是渲染效果，就必须在：

- 渲染流程内
- 正确阶段

被执行。

所以纯色输出虽然简单，但它仍然必须走正规渲染入口。

## 7. 第 4 件事：在入口里添加一个全屏 Pass

### 7.1 这里主要是谁负责

主要还是：

- `SceneViewExtension.cpp`

### 7.2 什么叫“添加一个全屏 Pass”

就是说：

**在这一帧里额外安排一个渲染步骤，对整张屏幕做一次处理。**

这里的处理非常简单：

- 不采样复杂纹理
- 不计算深度
- 不做法线判断

只做一件事：

- 给整张输出写一个固定颜色

### 7.3 为什么这一步一定要用 Pass 来理解

因为从这一步开始，你就要养成图形学工程思维：

- 不是“我执行一个函数画红色”
- 而是“我向渲染图里添加一个 Pass”

哪怕这个 Pass 很简单，  
它也已经是你以后所有效果的雏形了。

## 8. 第 5 件事：让这个 Pass 调用 Shader

### 8.1 这里主要是谁负责

涉及两部分：

- `SceneViewExtension.cpp`
- `.usf`

### 8.2 两边分别做什么

#### C++ 侧

主要负责：

- 指定这个 Pass 要用哪个 Shader
- 准备 Shader 参数
- 组织输入输出

#### Shader 侧

主要负责：

- 对当前像素输出固定颜色

### 8.3 这一步的意义

这是你第一次真正验证：

- “我的 Global Shader 真的被调用了”

如果你最后看到了整屏颜色，  
说明 GPU 侧代码确实已经参与工作了。

## 9. 纯色 Shader 本身需要做什么

这一步你先只理解概念，不急着看完整代码。

对这个最小效果来说，Shader 的任务非常简单：

### 9.1 输入

它可能只需要：

- 当前像素位置或 UV
- 一个固定颜色参数

### 9.2 输出

直接输出这个颜色。

也就是说，它不需要：

- 深度采样
- 法线采样
- 邻域卷积
- 多次采样

这就是为什么它特别适合作为第一个效果。

## 10. 纯色效果里 C++ 侧到底在忙什么

很多小白会误以为：

- 纯色输出这么简单，那 C++ 应该几乎不用干活

其实不是。

对这个效果来说，C++ 的工作量虽然不大，但职责非常关键：

### 10.1 建立工程链路

- 插件
- 模块
- 扩展

### 10.2 把 Shader 接进渲染流程

- 找到 Shader
- 绑定 Shader
- 给 Shader 参数

### 10.3 组织 Pass

- 告诉 RDG 这个 Pass 要干什么
- 输出写到哪里

所以你可以把这个阶段的 C++ 理解成：

**“铺路的人”**

而不是“真正计算颜色的人”。

## 11. 为什么这个效果对新手价值特别高

因为它非常擅长回答一个关键问题：

**“到底是我的渲染链没通，还是只是算法有问题？”**

如果是纯色输出：

- 没有复杂算法
- 没有复杂输入
- 没有深度重建

所以只要出问题，范围就会被大大缩小。

### 11.1 如果完全没看到颜色

优先怀疑：

- 插件没加载
- 模块没启动
- View Extension 没被回调
- Pass 没加进去
- Shader 没编译成功

### 11.2 如果颜色不对

优先怀疑：

- Shader 参数传递
- 输出目标
- 混合或覆盖方式

这就是它为什么是一个优秀的“链路验证器”。

## 12. 这个最小效果里，每个文件大概负责什么

再按文件维度总结一遍。

### 12.1 `.uplugin`

负责：

- 让 UE 认识这个插件

### 12.2 `Build.cs`

负责：

- 让渲染相关模块参与编译

### 12.3 `Module.cpp`

负责：

- 启动模块
- 注册 Shader 目录
- 创建 View Extension

### 12.4 `SceneViewExtension.cpp`

负责：

- 在每帧渲染时拿到入口
- 添加全屏 RDG Pass
- 绑定纯色 Shader

### 12.5 `.usf`

负责：

- 输出固定颜色

你可以发现：

**真正“画颜色”的代码其实最少，工程接入层反而占了大头。**

这也是图形学工程项目和普通算法题最大的区别之一。

## 13. 纯色效果完成后，你就证明了什么

只要你最后真的看到了纯色结果，  
你至少证明了下面这些事情大概率已经成立：

1. 插件结构没问题
2. 模块正确启动了
3. Shader 目录映射没问题
4. Scene View Extension 成功参与了渲染流程
5. RDG Pass 成功被添加
6. Global Shader 成功编译并运行
7. 结果成功写到了输出目标

这 7 件事一旦成立，后面你再做深度可视化会轻松很多。

## 14. 你现在应该怎么理解“代码层面要做哪几件事”

不要把它理解成：

- 我是不是要背很多 API

你现在更应该这样理解：

- 哪一层负责接入
- 哪一层负责入口
- 哪一层负责组织 Pass
- 哪一层负责真正计算

也就是：

- `Module` 负责接入
- `Scene View Extension` 负责入口和组织
- `Shader` 负责计算

这三个角色分清了，代码就不会显得那么乱。

## 15. 对你现在最重要的不是“代码细节”，而是“任务拆分感”

如果你一上来就直接看一大段渲染代码，  
很容易觉得：

- 什么都看不懂
- 代码都长得差不多
- 不知道哪段是关键

但如果你先把这个纯色效果拆成 5 件事：

1. 插件加载
2. 模块初始化
3. 进入每帧渲染
4. 加全屏 Pass
5. Shader 输出固定颜色

那你再看代码时，就会知道每段代码到底是在解决哪一件事。

## 16. 这篇看完后你至少应该能回答的问题

- 为什么纯色全屏输出不只是写一个 `.usf` 文件？
- 为什么 Module 必须参与这个效果？
- 为什么必须通过 `Scene View Extension` 才能在每帧渲染里插入逻辑？
- 为什么纯色效果也要用 RDG Pass？
- 为什么 Shader 虽然简单，但工程接入反而更关键？

## 17. 最后一句话总结

纯色全屏输出这个最小效果，代码层面本质上只是在做五件事：

**让插件被加载，让模块完成初始化，让扩展对象进入每帧渲染入口，在入口里添加一个全屏 RDG Pass，再让这个 Pass 调用一个输出固定颜色的 Shader。**
