# 最小可运行版本需要哪些文件 - 小白版

## 1. 这篇文档解决什么问题

前两篇你已经知道了：

- Plugin 是载体
- Module 会启动
- Scene View Extension 是入口
- RDG 用来组织 Pass
- Global Shader 在 GPU 上算效果

但如果现在让你真的开始做，你大概率会卡在这里：

**“那我到底要建哪些文件？”**

这篇就只回答这个问题。

目标不是做完整 SSAO，  
而是先明确：

**一个 UE5 RDG 渲染插件的第一个最小可运行版本，最少需要哪些文件。**

## 2. 什么叫“最小可运行版本”

这里的“最小可运行版本”不是指：

- 效果已经很好看
- 算法已经完整
- 可以写进简历

它只是指：

**插件能被 UE 正确加载，模块能启动，能注册 Scene View Extension，能找到 Shader 文件，并且具备插入 RDG Pass 的工程骨架。**

换句话说，这个阶段主要是在证明：

- 结构通了
- 链路通了
- 后面可以在这个壳子里继续加效果

## 3. 先看最终文件树

建议你第一个版本按这个结构来理解：

```text
Plugins/
└─ RDGStarter/
   ├─ RDGStarter.uplugin
   ├─ Source/
   │  └─ RDGStarter/
   │     ├─ RDGStarter.Build.cs
   │     ├─ Public/
   │     │  ├─ RDGStarterModule.h
   │     │  └─ RDGStarterSceneViewExtension.h
   │     └─ Private/
   │        ├─ RDGStarterModule.cpp
   │        └─ RDGStarterSceneViewExtension.cpp
   └─ Shaders/
      └─ Private/
         └─ RDGStarterFullscreen.usf
```

你现在不用纠结名字是不是最优，  
重点是理解：

- 为什么要有这些文件
- 少了哪个就会断链

## 4. 每个文件分别是干什么的

下面按“从外到内”的顺序讲。

## 5. `RDGStarter.uplugin` 是什么

### 5.1 作用

这个文件是插件的说明书。

它告诉 UE：

- 这是一个插件
- 插件名字是什么
- 有哪些模块
- 模块什么时候加载

### 5.2 没有它会怎样

没有 `.uplugin`，UE 根本不知道这是一个插件。

所以它的角色是：

**“插件的身份证 + 装配清单”**

### 5.3 你现在该怎么理解它

对你来说，这个文件不负责渲染效果，  
它只负责：

**把整个插件挂到 UE 识别体系里。**

## 6. `RDGStarter.Build.cs` 是什么

### 6.1 作用

这个文件是模块的编译配置文件。

它会告诉 UE 编译系统：

- 这个模块依赖哪些引擎模块
- 需要包含哪些功能
- 哪些头文件和库要参与编译

### 6.2 为什么它重要

你做普通 Gameplay 模块时，依赖可能主要是：

- Core
- Engine
- Input

但你做渲染插件时，通常还会涉及：

- RenderCore
- Renderer
- RHI
- Projects

所以这个文件本质是在说：

**“为了让这段代码能编译，我需要哪些引擎能力。”**

### 6.3 你现在该怎么理解它

把它理解成：

**“模块的编译菜单”**

## 7. `RDGStarterModule.h / RDGStarterModule.cpp` 是什么

### 7.1 作用

这是你的插件模块本体。

通常它会负责：

- 模块启动
- 模块关闭
- 初始化 Shader 目录映射
- 创建或注册 Scene View Extension

### 7.2 为什么最小版本一定要有它

因为插件不是自动会工作的。

你必须有一个地方做这些初始化动作：

- “我启动了”
- “Shader 放在哪”
- “我要注册扩展了”

这个“地方”通常就是 Module。

### 7.3 它在整条链里的位置

你可以把它理解成：

**插件的总控入口**

链路上它的作用是：

- Plugin 被 UE 识别
- Module 真正开始执行

### 7.4 这个文件先不用做太多事

第一个版本里，它只需要负责：

- 正常启动
- 能打印日志
- 能注册 Scene View Extension
- 能让 Shader 目录生效

不要一开始就在 Module 里堆太多逻辑。

## 8. `RDGStarterSceneViewExtension.h / .cpp` 是什么

### 8.1 作用

这是你真正接入渲染流程的核心文件。

它负责：

- 挂到 UE 的 View 渲染流程里
- 在每帧的某个阶段获得回调
- 在回调里插入你的 RDG Pass

### 8.2 为什么最小版本一定要有它

因为没有它，你虽然有插件、有模块、有 Shader：

- 但你的代码没有机会参与每帧渲染

所以它就是整条链里的：

**“渲染入口文件”**

### 8.3 小白视角怎么理解它

如果说 Module 是总控台，  
那 Scene View Extension 就是：

**“总控台里专门负责接渲染流程电话的人。”**

UE 每帧渲染时会“打电话给它”：

- 现在到你了
- 你要不要加点自己的渲染逻辑

### 8.4 这个文件以后会变得很重要

后面你写的很多内容都会逐渐堆到这里，包括：

- 读取 View 信息
- 构建 RDG 资源
- 添加 Pass
- 调用 Shader

所以这个文件以后会是整个图形学项目最核心的代码之一。

## 9. `Shaders/Private/RDGStarterFullscreen.usf` 是什么

### 9.1 作用

这是你的 Shader 文件。

它负责真正跑在 GPU 上，做图形计算。

### 9.2 为什么最小版本就要准备它

因为你的目标不是只做一个“空插件”，  
而是做一个“能参与渲染的插件”。

哪怕第一个版本的 Shader 很简单，比如：

- 输出纯色
- 可视化深度
- 显示一个调试结果

它也必须存在。

### 9.3 为什么放在 `Shaders/Private`

这是 UE 插件里常见的组织方式。

你现在先不用深挖“为什么叫 Private”，  
只要先记住：

- Shader 文件一般要放在插件自己的 `Shaders` 目录里
- 后面模块启动时会把这个目录映射给引擎识别

## 10. 这些文件之间是怎么串起来的

现在把它们按运行顺序串起来：

### 10.1 `.uplugin`

告诉 UE：

- 这里有个插件
- 插件里有个模块

### 10.2 `Build.cs`

告诉编译系统：

- 这个模块依赖哪些渲染相关模块

### 10.3 `Module.cpp`

模块启动时：

- 打日志
- 注册 Shader 目录
- 创建 Scene View Extension

### 10.4 `SceneViewExtension.cpp`

当 UE 每帧渲染时：

- 调用你的扩展回调
- 你在这里添加 RDG Pass

### 10.5 `.usf`

当 RDG Pass 执行时：

- GPU 跑这个 Shader
- 输出你想要的结果

这就是一条最小闭环。

## 11. 如果少了某个文件，会断在哪里

这个表很重要。

### 11.1 少了 `.uplugin`

断在：

- UE 根本识别不了你的插件

### 11.2 少了 `Build.cs`

断在：

- 模块无法正确编译

### 11.3 少了 `Module.cpp`

断在：

- 模块没有启动入口
- 没法注册 Shader 目录和 View Extension

### 11.4 少了 `SceneViewExtension.cpp`

断在：

- 插件无法进入每帧渲染流程

### 11.5 少了 `.usf`

断在：

- 你没有真正可运行的 GPU 效果代码

## 12. 第一个版本不需要哪些文件

这一步很关键。  
很多初学者一开始就建太多东西，把自己弄乱了。

第一个最小版本 **不需要** 这些：

- Editor 模块
- 设置面板
- 蓝图函数库
- 材质资产
- 复杂配置系统
- 多个 Shader 文件
- 多个效果开关
- 大量 Utility 类

也就是说，第一版先不要做：

- “工程设计很完整”
- “功能扩展很漂亮”

先只做一件事：

**把最小渲染链打通。**

## 13. 第一个版本真正的目标是什么

你不要把第一版目标理解成“做 SSAO”。

第一版真正的目标应该是：

**证明这个插件已经具备下面五件事：**

1. UE 能识别插件
2. 模块能启动
3. Shader 目录能注册成功
4. Scene View Extension 能参与每帧渲染
5. 至少一个最简单的 Shader 能被调用

只要这五件事成立，后面再往里加效果就会顺很多。

## 14. 第一个版本文件为什么不能再少

理论上某些头文件可以做合并，  
比如：

- 有些声明可以先全写在 cpp
- 文件名也可以更简单

但从“学习清晰度”来说，不建议再压缩。

因为你现在是小白阶段，  
最重要的不是文件越少越好，而是：

**每个角色分工尽量清楚。**

所以当前这套结构已经是：

- 足够小
- 但又不至于太乱

## 15. 你现在最应该形成的文件角色感

你以后看到这些文件，要在脑子里自动翻译成：

- `.uplugin`
  这是插件身份证

- `Build.cs`
  这是编译依赖配置

- `Module.cpp`
  这是插件启动总控

- `SceneViewExtension.cpp`
  这是渲染入口

- `.usf`
  这是 GPU 效果代码

如果你能做到这一点，后面真正开始写代码时就不会太慌。

## 16. 这一版文件结构对应的最小项目闭环

把整个最小闭环再压缩成一句话：

**插件描述文件让 UE 识别插件，模块启动后注册 Shader 和 Scene View Extension，扩展在每帧渲染时用 RDG 添加 Pass，Pass 调用 USF 里的 Global Shader 产出结果。**

## 17. 你看完这篇后，至少要能回答的问题

看完后你最好能回答：

- 为什么 `.uplugin` 和 `Build.cs` 不是一回事？
- 为什么模块启动文件必须存在？
- 为什么 Scene View Extension 是渲染入口文件？
- 为什么 Shader 文件不能省？
- 为什么第一版先不要加 Editor 模块和一堆配置？

## 18. 最后一句话总结

第一个最小可运行版本，真正必须有的不是很多：

**一个插件描述文件、一个模块配置文件、一个模块启动文件、一组 Scene View Extension 文件、一个最简单的 Shader 文件。**

只要这套骨架搭起来，你的 RDG 项目就真正有地方长内容了。
