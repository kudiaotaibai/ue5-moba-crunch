# Crunch 项目总结

## 项目概述

**项目名称：** Crunch

**项目类型：** 基于 Unreal Engine 5.5 的 MOBA 类多人对战游戏

**开发引擎：** Unreal Engine 5.5

**开发语言：** C++、Blueprint

**开发工具：** Visual Studio 2022、JetBrains Rider

**目标平台：** Windows 64位、Linux

---

## 一、核心技术架构

### 1.1 系统架构

```
┌─────────────────────────────────────────┐
│              客户端层                    │
│  (角色控制、UI、输入、特效渲染)          │
└─────────────────┬───────────────────────┘
                  │ EOS / Socket
┌─────────────────▼───────────────────────┐
│            游戏服务器层                  │
│  (游戏逻辑、状态同步、AI、技能计算)      │
└─────────────────┬───────────────────────┘
                  │ HTTP / Docker API
┌─────────────────▼───────────────────────┐
│            协调器层                      │
│  (服务器实例管理、端口分配、负载均衡)    │
└─────────────────────────────────────────┘
```

### 1.2 技术栈

| 类别 | 技术选型 |
|------|----------|
| 游戏引擎 | Unreal Engine 5.5 |
| 开发语言 | C++、Blueprint |
| 技能系统 | Gameplay Ability System (GAS) |
| 网络服务 | Epic Online Services (EOS) |
| 服务器部署 | Docker、Python Flask |
| 版本控制 | Git |
| 资源管理 | Asset Manager |

## 二、核心功能模块

### 2.1 游戏技能系统 (GAS)

**技术框架：** Gameplay Ability System

**核心组件：**

#### 基础属性集 (CAttributeSet)
- Health（生命值）
- MaxHealth（最大生命值）
- Mana（法力值）
- MaxMana（最大法力值）
- AttackDamage（攻击力）
- Armor（护甲）
- MoveSpeed（移速）

#### 英雄属性集 (CHeroAttributeSet)
- Intelligence（智力）
- Strength（力量）
- Experience（经验值）
- Level（等级）
- Gold（金币）
- UpgradePoint（技能点）
- StrengthGrowthRate（力量成长率）
- IntelligenceGrowthRate（智力成长率）

#### 技能类型
- **GA_Combo** - 连击系统
- **GA_GroundBlast** - 地面冲击（范围技能）
- **GA_Shoot** - 射击（投射物技能）
- **UpperCut** - 上勾拳（近战技能）

#### 状态效果
- **GAP_Dead** - 死亡状态
- **GAP_Launched** - 击飞状态
- 眩晕（Stun）
- 瞄准（Aim）

**特点：**
- 属性网络复制（Replication）
- 服务器权威验证
- 模块化技能设计
- 支持技能升级系统

---

### 2.2 角色系统

**基类：** ACCharacter

**核心功能：**

#### 角色类型
1. **玩家英雄角色**
   - 可控制的主角色
   - 完整的技能系统
   - 装备和物品系统

2. **AI 小兵 (Minion)**
   - 自动寻路
   - 目标选择
   - 团队识别
   - 根据队伍 ID 切换皮肤

#### 核心机制
- **团队系统**
  - 基于 IGenericTeamAgentInterface
  - 敌我识别
  - 队伍出生点分配

- **死亡与重生**
  - 死亡动画播放
  - 布娃娃物理效果
  - 重生机制
  - 死亡状态管理

- **状态管理**
  - 眩晕状态处理
  - 击飞效果
  - 瞄准状态切换

- **UI 系统**
  - 头顶血条显示
  - 可见性距离检测
  - 状态信息展示

---

### 2.3 AI 系统

**核心组件：**

#### 1. AI 控制器 (CAIController)
- 行为树执行
- 黑板数据管理
- 目标选择逻辑

#### 2. 小兵系统 (Minion)
- 自动寻路
- 目标追踪
- 攻击决策
- 队伍皮肤切换

#### 3. 兵营系统 (MinionBarrack)
- 定时生成小兵
- 目标分配
- 波次管理

#### 4. 行为树任务
- **BTTask_SendInputToAbilitySystem**
  - 将 AI 输入发送到技能系统
  - 触发技能释放

**特点：**
- 基于行为树的决策系统
- 黑板系统共享数据
- AI 感知组件
- 团队协作机制

---

### 2.4 网络系统

**架构模式：** 专用服务器（Dedicated Server）

#### Epic Online Services (EOS) 集成

**功能模块：**
1. **账号系统**
   - 账号门户登录
   - 客户端登录验证
   - 会话管理

2. **会话系统**
   - 创建游戏会话
   - 全局会话搜索
   - 会话加入
   - 会话 ID 匹配

3. **网络通信**
   - Socket 通信
   - 语音聊天（EOSVoiceChat）
   - 输入提供器（EOSOverlayInputProvider）

#### 服务器协调器

**技术栈：** Python Flask

**核心功能：**
- 动态创建游戏服务器实例
- 端口自动分配（从 7777 开始）
- HTTP API 接口
- 会话名称和搜索 ID 管理

**API 端点：**
```python
POST /Sessions
{
    "SESSION_NAME": "游戏会话名称",
    "SESSION_SEARCH_ID": "会话搜索ID"
}

Response:
{
    "status": "success",
    "PORT": 7777
}
```

#### Docker 部署

**架构：**
```yaml
services:
  server:          # 游戏服务器容器
  coordinator:     # 协调器容器
    ports:
      - '80:80'
```

**特点：**
- 容器化部署
- 自动化管理
- 端口映射
- 服务隔离

---

### 2.5 物品与商店系统

**核心组件：**

#### 1. 物品系统 (InventoryComponent)
- **容量管理**
  - 默认 6 格背包
  - 物品堆叠
  - 槽位管理

- **物品操作**
  - 购买物品
  - 出售物品
  - 使用物品
  - 物品合成

- **网络同步**
  - 服务器权威
  - 客户端通知
  - 物品状态复制

#### 2. 商店系统
- 物品购买
- 金币消耗
- 物品配方合成
- 装备效果应用

**数据资产：**
- **PA_ShopItem** - 商店物品定义
- **InventoryItem** - 物品实例

---

### 2.6 核心玩法机制

#### StormCore 推进系统

**机制：** 双方争夺影响区域的拔河式推进

**核心功能：**
1. **影响区域检测**
   - 球形碰撞检测
   - 队伍影响力计算
   - 动态目标更新

2. **推进逻辑**
   - 根据双方人数差异推进
   - 核心移动动画
   - 目标切换机制

3. **胜利条件**
   - 推进到对方基地
   - 触发胜利结算

**组件：**
- USphereComponent - 影响范围检测
- UDecalComponent - 地面标记
- UCameraComponent - 观察视角

## 五、开发经验总结

### 5.1 技术难点

#### 1. GAS 学习曲线陡峭
**问题：** 概念复杂，文档不够详细

**解决方案：**
- 从官方示例项目入手
- 逐步理解 Ability、Effect、Attribute
- 结合实际需求迭代实现

#### 2. 网络同步状态不一致
**问题：** 客户端和服务器状态不同步

**解决方案：**
- 采用服务器权威模式
- 关键逻辑在服务器执行
- 使用属性复制确保同步
- 客户端仅负责表现

#### 3. AI 行为不够智能
**问题：** 小兵容易卡住，行为单一

**解决方案：**
- 优化行为树结构
- 增加异常处理节点
- AI 感知系统动态更新目标
- 添加更多决策分支

#### 4. 服务器部署复杂
**问题：** 手动管理服务器实例繁琐

**解决方案：**
- 开发 Python 协调器自动化管理
- 采用 Docker 容器化简化部署
- HTTP API 统一管理接口

### 5.2 最佳实践

#### 代码组织
- 模块化设计，职责分离
- 使用接口实现多态
- 数据驱动配置
- 资源异步加载

#### 网络编程
- 服务器权威验证
- 最小化网络传输
- 客户端预测
- 延迟补偿

#### 性能优化
- 对象池复用
- LOD 层级管理
- 异步加载资源
- 减少 Tick 调用

#### 团队协作
- 代码规范统一
- 版本控制管理
- 文档及时更新
- 定期代码审查

## 九、项目总结

Crunch 是一个基于 Unreal Engine 5.5 的 MOBA 类多人对战游戏项目，成功实现了：

1. **完整的游戏技能系统** - 基于 GAS 框架，支持复杂的技能交互和状态管理
2. **稳定的多人网络架构** - 采用专用服务器模式，集成 EOS 在线服务
3. **智能的 AI 系统** - 基于行为树的小兵 AI，支持团队协作
4. **灵活的服务器部署** - Docker 容器化，Python 协调器自动管理

项目验证了 UE5 在多人竞技游戏开发中的可行性，为同类游戏开发提供了完整的技术方案和实践经验。
