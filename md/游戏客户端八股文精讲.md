## 第一部分：网络编程基础

## 一、TCP/UDP 协议深度解析

### 1.1 TCP 协议核心机制

#### 三次握手（建立连接）

**为什么需要三次握手？**

这是面试高频问题，要从多个角度理解：

**角度1：防止失效连接请求**
假设客户端发送了一个连接请求，但因为网络延迟，这个请求在网络中滞留了很久。客户端以为请求丢失，又发送了新的连接请求并成功建立连接。后来，第一个请求终于到达服务器。

- 如果是两次握手：服务器收到第一个请求就建立连接，但客户端已经不需要这个连接了，造成资源浪费。
- 三次握手：服务器回应后，客户端不会确认这个失效的请求，连接不会建立。

**角度2：确认双方收发能力**
- 第一次握手：服务器确认"客户端的发送能力正常"
- 第二次握手：客户端确认"服务器的收发能力正常"
- 第三次握手：服务器确认"客户端的接收能力正常"

**角度3：协商初始序列号**
TCP 是可靠传输，需要序列号来保证数据顺序和完整性。三次握手过程中，双方交换初始序列号（ISN），为后续数据传输做准备。

**游戏场景应用：**
在你的 Crunch 项目中，玩家连接游戏服务器时，TCP 三次握手确保连接可靠建立。如果握手失败，客户端可以提示"连接服务器失败，请检查网络"。

#### 四次挥手（断开连接）

**为什么需要四次挥手？**

**核心原因：TCP 是全双工通信**

全双工意味着双方可以同时发送和接收数据，所以关闭连接时，双方都要主动关闭自己的发送通道。

**详细过程：**
1. 客户端发送 FIN：我不再发送数据了
2. 服务器回应 ACK：我知道你不发了
3. 服务器发送 FIN：我也不发了（可能还有数据要发完）
4. 客户端回应 ACK：我知道你也不发了

**为什么不能三次？**
因为服务器收到客户端的 FIN 后，可能还有数据要发送。所以 ACK 和 FIN 要分开发送。

**TIME_WAIT 状态**

主动关闭方在发送最后一个 ACK 后，会进入 TIME_WAIT 状态，等待 2MSL（Maximum Segment Lifetime）时间。

**为什么需要 TIME_WAIT？**
1. 确保最后的 ACK 能到达对方。如果 ACK 丢失，对方会重发 FIN，TIME_WAIT 状态可以重新发送 ACK。
2. 让旧连接的数据包在网络中消失，避免影响新连接。

**游戏场景应用：**
玩家退出游戏时，客户端主动断开连接。如果服务器还有数据要发送（比如最后的战绩统计），四次挥手确保数据完整传输后再断开。

#### TCP 可靠性保证机制

**1. 序列号与确认应答**

每个字节都有序列号，接收方收到数据后发送 ACK（Acknowledgment，确认应答）确认。

**通俗理解：**
就像你给朋友发了 10 条微信，朋友收到后回复"我收到了前 8 条"，你就知道第 9、10 条可能丢了，需要重发。

**具体例子：**
```
发送方：发送 seq=100, len=100 的数据（字节 100-199）
接收方：回应 ACK=200（表示 200 之前的都收到了，期待从 200 开始的数据）
```

**2. 超时重传**

发送数据后启动定时器，如果超时未收到 ACK（确认应答），重传数据。

**RTO（Retransmission Timeout，重传超时时间）计算：**
- 不能太短：会导致不必要的重传（其实数据还在路上，只是慢了点）
- 不能太长：影响传输效率（数据真丢了，等太久才重传）

TCP 使用动态 RTO，根据网络状况自适应调整。

**通俗理解：**
就像你给朋友发微信，如果 5 秒没回复，你可能会再发一次。但如果朋友网络慢，你 1 秒就重发，就太频繁了；如果等 1 分钟才重发，又太慢了。TCP 会根据网络情况自动调整这个等待时间。

**3. 流量控制（滑动窗口）**

接收方通告窗口大小，发送方根据窗口大小控制发送速率。

**什么是窗口？**
窗口就是"一次可以发送多少数据"的限制。就像水管的粗细决定了水流速度。

**滑动窗口机制：**
```
发送窗口：[已发送已确认][已发送未确认][可以发送][不能发送]
接收窗口：[已接收已确认][可以接收][不能接收]
```

**通俗理解：**
你有 100 个快递要发，但快递公司说"你一次最多给我 10 个"，这个 10 就是窗口大小。发出去 10 个后，收到确认才能再发下一批。

**游戏场景：**
如果玩家网络带宽有限（比如用手机流量），接收窗口会变小，服务器自动降低发送速率，避免丢包。

**4. 拥塞控制**

防止网络过载，TCP 有四个算法：

**慢启动（Slow Start）：**
- 初始窗口很小（通常 1-10 个 MSS，MSS = Maximum Segment Size，最大报文段大小，通常是 1460 字节）
- 每收到一个 ACK（确认应答），窗口大小翻倍（指数增长）
- 快速探测网络容量

**通俗理解：**
就像你第一次去一个新餐厅，不知道能吃多少，先点一份试试。如果吃得下，下次点两份，再下次点四份，快速找到自己的食量。

**拥塞避免（Congestion Avoidance）：**
- 达到阈值后，改为线性增长
- 每个 RTT（Round Trip Time，往返时间，数据发出去再收到回复的时间）窗口增加 1 个 MSS
- 缓慢增加，避免拥塞

**通俗理解：**
找到食量后，就不能再翻倍加了，而是每次只多加一点点，避免吃撑。

**快速重传（Fast Retransmit）：**
- 收到 3 个重复 ACK（确认应答），立即重传
- 不等超时，提高效率

**通俗理解：**
如果朋友连续 3 次说"我没收到你的第 5 条消息"，你就知道第 5 条消息肯定丢了，立即重发，不用等超时。

**快速恢复（Fast Recovery）：**
- 快速重传后，窗口减半而不是降到最小
- 继续传输，不重新慢启动

**通俗理解：**
发现吃撑了（网络拥塞），不是完全不吃了（重新慢启动），而是减半食量继续吃（窗口减半继续传输）。

**游戏场景应用：**
玩家刚连接服务器时，TCP 慢启动逐渐增加发送速率。如果网络拥塞（丢包），自动降低速率。这就是为什么有时候刚进游戏会有点卡，过一会就流畅了。

### 1.2 UDP 协议特性

#### UDP 的优势

**1. 无连接**
- 不需要建立连接，直接发送
- 减少延迟

**通俗理解：**
TCP 像打电话，要先拨号等对方接听才能说话；UDP 像发短信，写完直接发，不管对方是否在线。

**2. 不可靠**
- 不保证到达（可能丢失）
- 不保证顺序（可能乱序）
- 不保证完整性（可能损坏）

**通俗理解：**
就像寄平信，邮局不保证一定送到，也不保证按顺序送到。

**3. 低开销**
- 首部只有 8 字节（TCP 需要 20 字节）
- 没有拥塞控制
- 没有流量控制

**通俗理解：**
UDP 就像明信片，地址写得简单；TCP 像挂号信，要填很多表格。

**4. 支持广播和组播**
- 一对多通信
- 适合实时通信

**通俗理解：**
UDP 可以像广播电台一样，一个人说话，很多人同时听到。

#### 为什么游戏常用 UDP？

**实时性优先：**
游戏中，玩家位置、动作等信息需要实时传输。丢失一帧数据不要紧，下一帧会更新。但如果等待重传，会造成明显延迟。

**通俗理解：**
看电视直播时，偶尔花屏一下没关系，但如果为了保证画面完美而延迟 5 秒播出，你就看不了直播了。

**举例：**
- 玩家 A 在位置 (100, 200)
- 0.1 秒后移动到 (110, 210)
- 如果第一个位置包丢失，不需要重传，直接发送新位置即可

**TCP 的问题：**
如果某个包丢失，TCP 会阻塞后续所有数据，等待重传。这在游戏中是不可接受的。

**通俗理解：**
TCP 就像严格的老师，第 5 道题没做对，就不让你做第 6 道题；UDP 像宽松的老师，第 5 道题错了没关系，继续做第 6 道题。

**UDP 的应用场景：**
- 玩家位置同步（实时性重要）
- 动作同步（偶尔丢失无所谓）
- 语音聊天（实时对话）
- 实时对战（快速响应）

**TCP 的应用场景：**
- 登录验证（必须准确）
- 聊天消息（不能丢失）
- 物品交易（涉及金钱，必须可靠）
- 关键操作（技能学习、装备强化等）

### 1.3 如何在 UDP 上实现可靠传输？

这是一个高级问题，展示你的深度理解。

**核心思路：在应用层实现 TCP 的部分机制**

**什么是应用层？**
网络分为多层，从下到上是：物理层 → 数据链路层 → 网络层 → 传输层 → 应用层。TCP/UDP 在传输层，我们的游戏程序在应用层。

**1. 添加序列号**
```
struct UDPPacket {
    uint32_t sequence;  // 序列号（这是第几个包）
    uint32_t ack;       // 确认号（我收到了第几个包）
    uint32_t ackBits;   // 位域，表示最近 32 个包的接收情况
    byte data[];        // 实际数据
}
```

**通俗理解：**
给每个包编号，就像给快递编号一样，这样就知道哪个包丢了。

**2. 确认机制**
- 接收方收到包后，记录序列号
- 在下一个发送的包中，携带 ACK（确认应答）信息
- 使用位域表示最近收到的包

**通俗理解：**
你收到朋友的第 5 条消息，在回复第 6 条消息时顺便说"我收到了你的第 5 条消息"。

**3. 超时重传**
- 记录每个包的发送时间
- 如果一定时间内没有收到 ACK（确认应答），重传
- 只重传重要的包（不是所有包都需要可靠）

**通俗理解：**
发了重要消息，如果朋友 10 秒没回复，就再发一次。但发的只是闲聊，就不用重发了。

**4. 选择性重传**
- 不是所有数据都需要可靠传输
- 标记哪些包需要可靠，哪些不需要
- 位置数据：不可靠（丢了就丢了，下一帧会更新）
- 技能释放：可靠（必须确保收到）

**实际案例：QUIC 协议**
Google 开发的 QUIC 协议就是在 UDP 上实现了可靠传输，HTTP/3 使用 QUIC。

**游戏场景应用：**
在你的项目中，可以这样设计：
- 玩家移动：UDP 不可靠传输（实时性重要）
- 技能释放：UDP + 应用层确认（既要快又要准）
- 物品交易：TCP 可靠传输（绝对不能出错）

## 二、网络同步机制

### 2.1 网络延迟的本质

**延迟的组成：**
```
总延迟 = 传播延迟 + 传输延迟 + 处理延迟 + 排队延迟
```

**1. 传播延迟（Propagation Delay）**
- 信号在介质中传播的时间
- 光速约 200,000 km/s（光纤中的光速比真空中慢）
- 北京到上海约 1000km，传播延迟约 5ms
- 这是物理极限，无法优化

**通俗理解：**
就像你喊话，声音传到对方耳朵需要时间。距离越远，时间越长，这是物理定律，无法改变。

**2. 传输延迟（Transmission Delay）**
- 数据包从网卡发送出去的时间
- 取决于带宽和数据包大小
- 1Mbps 带宽发送 1KB 数据：8ms

**通俗理解：**
就像你说一句话需要时间。话越长，说完需要的时间越长。网速越慢，发送数据需要的时间越长。

**3. 处理延迟（Processing Delay）**
- 路由器、交换机处理数据包的时间
- 通常很小（微秒级）

**通俗理解：**
快递员看地址、分拣包裹需要时间，但通常很快。

**4. 排队延迟（Queuing Delay）**
- 数据包在路由器队列中等待的时间
- 网络拥塞时会很大
- 最不稳定的部分

**通俗理解：**
就像快递站排队，人少时不用等，人多时要排很久队。

**游戏中的延迟表现：**
- Ping 值：往返时间（RTT，Round Trip Time）
- 50ms 以下：优秀（感觉不到延迟）
- 50-100ms：良好（轻微延迟）
- 100-200ms：一般（明显延迟）
- 200ms 以上：较差（严重影响游戏体验）

### 2.2 客户端预测（Client-Side Prediction）

**问题：**
如果玩家按下"前进"键，等待服务器响应再移动，会有明显延迟（100ms+），手感很差。

**通俗理解：**
就像你在网上购物，如果每次点击都要等服务器确认才能看到效果，网页就会很卡。

**解决方案：客户端预测**

**核心思想：**
客户端立即执行玩家的输入，不等服务器确认。

**通俗理解：**
你按下"前进"键，角色立即移动，同时告诉服务器"我要前进"。如果服务器说"可以"，就继续；如果说"不行"，就回到原位。

**详细流程：**

**1. 玩家输入**
```
时刻 T0：玩家按下 W 键
```

**2. 客户端立即执行**
```
时刻 T0：
- 客户端立即移动角色
- 记录这个输入命令：{commandID: 1, input: "forward", timestamp: T0}
- 发送给服务器
```

**通俗理解：**
你在餐厅点菜，不等服务员确认，先在心里想象这道菜的味道，同时告诉服务员你要点什么。

**3. 服务器验证**
```
时刻 T0 + 50ms（网络延迟）：
- 服务器收到命令
- 验证合法性（速度是否正常、是否在地图内、是否撞墙）
- 执行移动
- 返回确认和实际位置
```

**通俗理解：**
服务员检查你点的菜是否有货，价格是否正确，然后告诉你"可以"或"不行"。

**4. 客户端校正**
```
时刻 T0 + 100ms：
- 客户端收到服务器位置
- 对比本地位置和服务器位置
- 如果差异小：忽略
- 如果差异大：回滚并重放
```

**通俗理解：**
如果服务员说"这道菜没有了"，你就要重新选择；如果说"可以"，你就继续等菜。

**回滚重放机制：**

假设客户端预测错误（比如撞墙了但客户端没检测到）：

```
1. 客户端当前位置：(150, 200)
2. 服务器返回位置：(100, 200)
3. 差异过大（50 单位）

回滚重放：
- 将角色位置设置为服务器位置 (100, 200)
- 重新执行所有未确认的命令
- 命令 2：前进 -> (110, 200)
- 命令 3：前进 -> (120, 200)
- 最终位置：(120, 200)
```

**通俗理解：**
你以为可以穿过一堵墙，结果撞墙了。服务器告诉你"你在墙前面"，你就回到墙前面，然后重新执行后续的动作（比如绕过墙）。

**优化：误差阈值**

不是所有差异都需要回滚，设置阈值：
```
if (distance > 100.0f) {
    // 差异过大，回滚（立即纠正）
    Rollback();
} else if (distance > 10.0f) {
    // 差异较小，平滑插值（慢慢纠正）
    SmoothCorrect();
} else {
    // 差异很小，忽略
    Ignore();
}
```

**通俗理解：**
如果你的位置和服务器差得太多，立即纠正；差得不多，慢慢调整；差得很少，就不管了。

**游戏场景应用：**
在你的 Crunch 项目中：
- 玩家移动：客户端预测（按键立即响应）
- 技能释放：客户端预测（播放动画），服务器验证（扣除蓝量和设置 CD）
- 伤害计算：服务器权威（防止作弊）

**什么是服务器权威？**
重要的计算由服务器做决定，客户端只负责显示。就像考试时，你可以先在草稿纸上写答案（客户端预测），但最终分数由老师判定（服务器权威）。

### 2.3 服务器校正（Server Reconciliation）

**问题：**
客户端预测可能出错，需要服务器校正。

**什么是校正？**
就是纠正错误。如果你预测错了，服务器会告诉你正确的结果。

**校正策略：**

**1. 硬校正（Hard Correction）**
```
直接将客户端位置设置为服务器位置
```
- 优点：简单，保证一致
- 缺点：会有瞬移感，体验差

**通俗理解：**
就像你走错路了，直接瞬间传送到正确位置。虽然准确，但感觉很突兀。

**2. 软校正（Soft Correction）**
```
平滑插值到服务器位置
```
- 优点：体验好，无瞬移
- 缺点：短时间内位置不准确

**通俗理解：**
走错路了，慢慢走回正确位置。感觉自然，但需要时间。

**3. 混合策略**
```
if (误差 > 阈值1) {
    硬校正（瞬移）
} else if (误差 > 阈值2) {
    软校正（插值）
} else {
    忽略
}
```

**通俗理解：**
如果错得太离谱，直接瞬移；错得不多，慢慢调整；错得很少，不管。

**实现软校正：**
```
每帧执行：
    当前位置 = Lerp(当前位置, 服务器位置, 校正速度 * DeltaTime)
    
    if (距离服务器位置很近) {
        停止校正
    }
```

**什么是 Lerp？**
Lerp 是 Linear Interpolation（线性插值）的缩写，就是在两个值之间平滑过渡。

**什么是 DeltaTime？**
DeltaTime 是两帧之间的时间间隔，通常是 1/60 秒（60FPS 时）。

**通俗理解：**
每帧都朝着正确位置移动一点点，就像慢慢调整方向盘一样。

**校正速度的选择：**
- 太快：会有抖动感
- 太慢：长时间不准确
- 建议：根据误差大小动态调整

```
校正速度 = 基础速度 * (误差 / 最大误差)
```

**通俗理解：**
错得越多，调整得越快；错得越少，调整得越慢。

### 2.4 延迟补偿（Lag Compensation）

**问题：射击游戏中的命中判定**

玩家 A 看到玩家 B 在位置 X，开枪射击。但由于网络延迟，服务器上玩家 B 已经移动到位置 Y。如果服务器用当前位置判定，玩家 A 明明瞄准了却打不中。

**通俗理解：**
你看到敌人在门口，开枪射击，但因为网络延迟，敌人实际上已经跑到房间里了。如果按敌人的实际位置判定，你明明瞄准了却打不中，这很不公平。

**解决方案：服务器回溯（Lag Compensation）**

**核心思想：**
服务器回溯到玩家 A 开枪时看到的世界状态，在那个状态下判定命中。

**通俗理解：**
服务器说："让我看看玩家 A 开枪那一刻，敌人确实在哪里。"然后在那个时刻的位置判定是否命中。

**详细实现：**

**1. 服务器保存历史快照**
```
每 50ms 保存一次所有玩家的位置快照：
快照 1：时间 T0，玩家 B 位置 (100, 200)
快照 2：时间 T0+50ms，玩家 B 位置 (110, 210)
快照 3：时间 T0+100ms，玩家 B 位置 (120, 220)
...
保留最近 1 秒的快照（20 个）
```

**通俗理解：**
就像监控录像，每隔一段时间拍一张照片，记录每个人在哪里。

**2. 客户端发送开枪请求**
```
{
    action: "shoot",
    direction: (1, 0, 0),
    clientTime: T0+200ms
}
```

**通俗理解：**
玩家告诉服务器："我在 T0+200ms 这个时刻，朝着 (1,0,0) 方向开枪了。"

**3. 服务器计算延迟**
```
服务器时间：T0+300ms
客户端时间：T0+200ms
延迟：100ms
```

**通俗理解：**
服务器算出："这个玩家的消息晚到了 100ms。"

**4. 服务器回溯**
```
回溯到时间：T0+300ms - 100ms = T0+200ms
在快照中找到 T0+200ms 时刻的世界状态
玩家 B 在那个时刻的位置：(115, 215)（插值计算）
```

**通俗理解：**
服务器翻看录像，找到玩家开枪那一刻，敌人确实在哪里。

**5. 在回溯的世界中判定**
```
从开枪位置沿着方向做射线检测
如果射线击中玩家 B 的历史位置，判定命中
```

**通俗理解：**
在那个时刻的世界里，看看子弹是否能打中敌人。

**6. 应用伤害**
```
在当前世界状态中，对玩家 B 造成伤害
```

**通俗理解：**
如果判定命中，就在现在的世界里扣血。

**防作弊限制：**

不能无限回溯，否则玩家可以修改客户端时间作弊：

```
最大回溯时间 = 1 秒

if (延迟 > 1秒) {
    拒绝请求，可能是作弊
}
```

**通俗理解：**
不能让玩家说"我是 10 分钟前开的枪"，那样就可以打到早就跑掉的敌人了。

**游戏场景应用：**
- 射击游戏：必须使用延迟补偿（CS:GO、PUBG 都用这个技术）
- MOBA 游戏：技能命中判定可以使用
- 格斗游戏：攻击判定可以使用

**权衡：**
- 对射击者公平：看到就能打中
- 对被击者不公平：明明躲开了还被打中
- 需要在两者之间平衡

**通俗理解：**
这就像裁判的判罚，不可能让所有人都满意，只能尽量公平。

### 2.5 插值与外推（Interpolation & Extrapolation）

**问题：其他玩家的移动同步**

服务器每秒发送 20 次位置更新（50ms 一次），但游戏是 60fps（16.6ms 一帧）。如何让其他玩家的移动看起来流畅？

**什么是 fps？**
FPS = Frames Per Second，每秒帧数。60fps 意思是每秒显示 60 张画面，每张画面间隔 16.6ms。

**通俗理解：**
就像看电影，电影是 24fps（每秒 24 张图片），但网络更新只有 20fps。如何让角色移动看起来不卡顿？

**解决方案 1：插值（Interpolation）**

**核心思想：**
在两个已知位置之间平滑过渡。

**什么是插值？**
插值就是在两个已知点之间填充中间值。就像你知道 1 和 3，可以推算出中间是 2。

**实现：**
```
收到位置更新：
    位置 A：时间 T0，坐标 (100, 200)
    位置 B：时间 T0+50ms，坐标 (110, 210)

渲染时（T0+25ms）：
    进度 = (当前时间 - T0) / (T0+50ms - T0) = 0.5
    当前位置 = Lerp(位置A, 位置B, 进度)
             = (105, 205)
```

**通俗理解：**
你知道朋友 0 秒时在 A 点，50ms 时在 B 点，那么 25ms 时他应该在 A 和 B 的中间。

**优点：**
- 移动流畅
- 不会出现错误预测

**缺点：**
- 显示的是过去的位置（延迟 50ms）
- 对于快速移动的对象，会有滞后感

**解决方案 2：外推（Extrapolation）**

**核心思想：**
根据速度和方向，预测未来位置。

**什么是外推？**
外推就是根据已知信息推测未知信息。就像你看到朋友朝北走，可以推测他接下来还会继续朝北走。

**实现：**
```
最后收到的位置：(100, 200)
速度：(10, 10) 单位/秒
时间差：30ms

预测位置 = 最后位置 + 速度 * 时间差
         = (100, 200) + (10, 10) * 0.03
         = (100.3, 200.3)
```

**通俗理解：**
你知道朋友最后在 A 点，以每秒 10 米的速度朝北走，那么 0.03 秒后他应该在 A 点北边 0.3 米的地方。

**优点：**
- 显示的是当前或未来位置
- 响应更快

**缺点：**
- 预测可能错误（玩家突然转向）
- 需要校正，可能有抖动

**混合策略：**

```
if (速度稳定 && 方向稳定) {
    使用外推
} else {
    使用插值
}
```

**通俗理解：**
如果朋友一直直线走，就预测他继续直线走；如果他经常转向，就用保守的插值方法。

**游戏场景应用：**
- 匀速移动的对象：外推（小兵、子弹）
- 玩家控制的角色：插值（更安全，因为玩家操作不可预测）
- 快速移动的对象：插值（避免预测错误）

## 三、客户端预测与服务器校正

### 3.1 预测-校正模型的完整流程

让我们通过一个完整的例子，串联所有概念：

**场景：玩家 A 向前移动**

**时间线：**

```
T0 时刻（客户端）：
├─ 玩家按下 W 键
├─ 客户端立即移动角色：位置 (100, 200) -> (110, 200)
├─ 记录命令：{ID: 1, input: "forward", time: T0}
└─ 发送给服务器

T0+50ms（服务器）：
├─ 收到命令
├─ 验证合法性
├─ 执行移动：位置 (100, 200) -> (110, 200)
├─ 广播给其他玩家
└─ 返回确认给玩家 A

T0+100ms（客户端）：
├─ 收到服务器确认
├─ 服务器位置：(110, 200)
├─ 客户端位置：(120, 200)（期间又移动了）
├─ 对比差异：10 单位
├─ 差异在阈值内，忽略
└─ 移除已确认的命令

T0+50ms（其他玩家客户端）：
├─ 收到玩家 A 的位置更新
├─ 上一个位置：(100, 200)
├─ 新位置：(110, 200)
├─ 使用插值平滑移动
└─ 每帧：位置 = Lerp(旧位置, 新位置, 进度)
```

**如果预测错误：**

```
T0 时刻（客户端）：
├─ 玩家按下 W 键
├─ 客户端预测：位置 (100, 200) -> (110, 200)
└─ 发送给服务器

T0+50ms（服务器）：
├─ 收到命令
├─ 验证：前方有墙，不能移动
├─ 位置保持：(100, 200)
└─ 返回实际位置

T0+100ms（客户端）：
├─ 收到服务器位置：(100, 200)
├─ 客户端位置：(120, 200)（预测移动了两次）
├─ 差异：20 单位，超过阈值
├─ 回滚：位置设置为 (100, 200)
├─ 重放未确认的命令（命令 2）
└─ 最终位置：(100, 200)（因为撞墙）
```

### 3.2 不同游戏类型的同步策略

**FPS 射击游戏：**
- 玩家移动：客户端预测 + 服务器校正
- 射击判定：服务器回溯
- 子弹：客户端预测轨迹，服务器判定命中
- 优先级：响应速度 > 一致性

**MOBA 游戏：**
- 玩家移动：客户端预测 + 服务器校正
- 技能释放：客户端预表现 + 服务器验证
- 小兵：服务器权威，客户端插值
- 优先级：一致性 > 响应速度

**格斗游戏：**
- 输入：延迟输入（Delay-based）或回滚（Rollback）
- 判定：服务器权威
- 优先级：公平性 > 响应速度

**MMO 游戏：**
- 玩家移动：客户端预测 + 服务器校正
- 战斗：服务器权威
- 非战斗交互：客户端预测
- 优先级：一致性 > 响应速度

## 第二部分：游戏架构设计

## 四、客户端-服务器架构

### 4.1 架构模式对比

#### 1. P2P（点对点）架构

**特点：**
- 没有专用服务器
- 玩家之间直接通信
- 一个玩家作为 Host

**优点：**
- 无需服务器成本
- 延迟低（直连）

**缺点：**
- Host 玩家退出游戏结束
- 容易作弊（Host 可以修改数据）
- 网络拓扑复杂

**适用场景：**
- 小规模对战（2-4人）
- 休闲游戏
- 本地联机

#### 2. 专用服务器（Dedicated Server）架构

**特点：**
- 独立的服务器程序
- 服务器权威
- 客户端只负责显示和输入

**优点：**
- 防作弊（服务器验证）
- 稳定性高
- 支持大规模对战

**缺点：**
- 服务器成本高
- 延迟增加（多一跳）

**适用场景：**
- 竞技游戏
- MMO 游戏
- 大规模对战

**你的项目使用的就是这种架构。**

#### 3. Listen Server 架构

**特点：**
- 一个玩家既是客户端又是服务器
- 其他玩家连接到这个玩家

**优点：**
- 无需专用服务器
- Host 玩家延迟为 0

**缺点：**
- Host 玩家有优势
- Host 退出游戏结束

**适用场景：**
- 合作游戏
- 小规模对战

### 4.2 服务器权威 vs 客户端权威

**服务器权威（Server Authoritative）：**

**核心原则：**
- 所有重要逻辑在服务器执行
- 客户端只发送输入
- 服务器验证并广播结果

**示例：技能释放**
```
客户端：
1. 玩家按下技能键
2. 播放技能动画（预表现）
3. 发送请求：ServerCastSkill(SkillID)

服务器：
1. 收到请求
2. 验证：CD 是否好了？蓝量够吗？目标在范围内吗？
3. 如果验证通过：
   - 扣除蓝量
   - 设置 CD
   - 计算伤害
   - 广播给所有客户端
4. 如果验证失败：
   - 通知客户端取消技能

其他客户端：
1. 收到广播
2. 播放技能特效
3. 显示伤害数字
```

**优点：**
- 防作弊
- 数据一致
- 公平性高

**缺点：**
- 延迟影响体验
- 服务器压力大

**客户端权威（Client Authoritative）：**

**核心原则：**
- 客户端计算结果
- 服务器只转发

**示例：**
```
客户端：
1. 玩家移动
2. 客户端计算新位置
3. 发送新位置给服务器

服务器：
1. 收到位置
2. 直接广播给其他客户端（不验证）
```

**优点：**
- 响应快
- 服务器压力小

**缺点：**
- 容易作弊
- 数据不一致

**适用场景：**
- 单机游戏
- 非竞技游戏
- 信任客户端的场景

**混合策略：**

实际游戏中，通常混合使用：

```
服务器权威：
- 战斗伤害
- 物品获得
- 金币交易
- 技能 CD

客户端权威（服务器宽松验证）：
- 玩家移动（服务器检查速度是否合理）
- 视角旋转
- 动画播放
```

### 4.3 RPC（远程过程调用）

**什么是 RPC？**

RPC 让你可以像调用本地函数一样调用远程函数。

**Unreal Engine 中的 RPC：**

**1. Server RPC（客户端调用，服务器执行）**
```cpp
UFUNCTION(Server, Reliable)
void ServerCastSkill(int32 SkillID);

// 客户端调用
ServerCastSkill(1);

// 服务器执行
void AMyCharacter::ServerCastSkill_Implementation(int32 SkillID) {
    // 在服务器上执行
    CastSkill(SkillID);
}
```

**2. Client RPC（服务器调用，客户端执行）**
```cpp
UFUNCTION(Client, Reliable)
void ClientShowDamage(float Damage);

// 服务器调用
ClientShowDamage(100.0f);

// 客户端执行
void AMyCharacter::ClientShowDamage_Implementation(float Damage) {
    // 在客户端上执行
    ShowDamageNumber(Damage);
}
```

**3. Multicast RPC（服务器调用，所有客户端执行）**
```cpp
UFUNCTION(NetMulticast, Reliable)
void MulticastPlayEffect(FVector Location);

// 服务器调用
MulticastPlayEffect(GetActorLocation());

// 所有客户端执行
void AMyCharacter::MulticastPlayEffect_Implementation(FVector Location) {
    // 在所有客户端上执行
    SpawnEffect(Location);
}
```

**Reliable vs Unreliable：**

**Reliable（可靠）：**
- 保证到达
- 保证顺序
- 用于重要操作

**Unreliable（不可靠）：**
- 不保证到达
- 不保证顺序
- 用于频繁更新

**使用建议：**
```
Reliable：
- 技能释放
- 物品使用
- 聊天消息

Unreliable：
- 位置更新
- 动画同步
- 特效播放
```

**RPC 的限制：**

1. 不能有返回值
2. 参数必须可序列化
3. 频繁调用会增加网络负担

**游戏场景应用：**
在你的 Crunch 项目中：
- 玩家释放技能：Server RPC
- 显示伤害数字：Client RPC
- 播放技能特效：Multicast RPC

## 五、状态同步 vs 帧同步

### 5.1 状态同步（State Synchronization）

**核心思想：**
同步游戏对象的状态（位置、血量、属性等）。

**工作流程：**
```
服务器：
1. 执行游戏逻辑
2. 计算所有对象的状态
3. 将状态变化发送给客户端

客户端：
1. 接收状态更新
2. 更新本地对象
3. 插值平滑显示
```

**优点：**
- 实现简单
- 容错性好（丢包影响小）
- 支持中途加入

**缺点：**
- 网络流量大
- 状态多时开销大
- 需要插值处理

**优化技术：**

**1. 增量同步**
只同步变化的状态，不同步未变化的。

**2. 优先级同步**
重要对象高频同步，不重要对象低频同步。

**3. 兴趣管理（AOI）**
只同步玩家视野内的对象。

**4. 压缩**
压缩数据减少流量。

**适用场景：**
- FPS 游戏
- MOBA 游戏
- MMO 游戏
- 你的 Crunch 项目

### 5.2 帧同步（Lockstep）

**核心思想：**
同步玩家的输入，所有客户端执行相同的逻辑。

**工作流程：**
```
每一帧：
1. 所有客户端收集本地输入
2. 将输入发送给服务器
3. 服务器收集所有输入
4. 服务器广播所有输入给所有客户端
5. 所有客户端执行相同的逻辑
```

**关键要求：**
- 所有客户端必须同步执行
- 逻辑必须完全确定性（相同输入 -> 相同输出）
- 不能有随机数、浮点误差

**优点：**
- 网络流量小（只传输输入）
- 天然防作弊（服务器可以重放验证）
- 支持录像回放

**缺点：**
- 延迟高（等待所有输入）
- 实现复杂（确定性要求高）
- 不支持中途加入
- 一个玩家卡顿影响所有人

**确定性问题：**

**问题 1：随机数**
```
错误：
float damage = Random(10, 20);  // 每个客户端结果不同

正确：
uint32 seed = frame_number;
float damage = SeededRandom(seed, 10, 20);  // 相同种子，相同结果
```

**问题 2：浮点误差**
```
错误：
position += velocity * deltaTime;  // 不同平台可能有误差

正确：
使用定点数或整数运算
```

**问题 3：遍历顺序**
```
错误：
for (auto& obj : unordered_set) {  // 顺序不确定
    obj.Update();
}

正确：
for (auto& obj : sorted_vector) {  // 顺序确定
    obj.Update();
}
```

**适用场景：**
- RTS 游戏（星际争霸、魔兽争霸）
- MOBA 游戏（王者荣耀）
- 回合制游戏

### 5.3 两种方案对比

| 特性 | 状态同步 | 帧同步 |
|------|---------|--------|
| 网络流量 | 大 | 小 |
| 延迟 | 低 | 高 |
| 实现难度 | 简单 | 复杂 |
| 确定性要求 | 低 | 高 |
| 中途加入 | 支持 | 困难 |
| 防作弊 | 需要服务器验证 | 天然防作弊 |
| 录像回放 | 困难 | 简单 |
| 适用游戏 | FPS、MMO | RTS、MOBA |

**选择建议：**
- 动作游戏、射击游戏：状态同步
- 策略游戏、回合制：帧同步
- MOBA：可以两者结合

## 六、ECS 架构

### 6.1 传统 OOP vs ECS

**传统 OOP（面向对象）：**

```
class GameObject {
    Transform transform;
    Renderer renderer;
    Collider collider;
    
    void Update() {
        // 更新逻辑
    }
}

class Player : GameObject {
    Health health;
    Inventory inventory;
    
    void Update() {
        // 玩家逻辑
    }
}
```

**问题：**
1. 继承层级复杂
2. 组件耦合
3. 缓存不友好（数据分散）

**ECS（Entity-Component-System）：**

**Entity（实体）：**
- 只是一个 ID
- 没有数据和逻辑

**Component（组件）：**
- 只有数据
- 没有逻辑

**System（系统）：**
- 只有逻辑
- 处理特定组件

**示例：**
```
Entity: Player (ID: 1)

Components:
- TransformComponent: {position: (100, 200), rotation: 0}
- HealthComponent: {current: 100, max: 100}
- VelocityComponent: {x: 10, y: 0}

Systems:
- MovementSystem: 处理所有有 Transform 和 Velocity 的实体
- RenderSystem: 处理所有有 Transform 和 Sprite 的实体
- HealthSystem: 处理所有有 Health 的实体
```

**ECS 的优势：**

**1. 灵活组合**
```
飞行敌人 = Transform + Sprite + Health + Velocity + AI
地面敌人 = Transform + Sprite + Health + AI
静态障碍物 = Transform + Sprite + Collider
```

**2. 缓存友好**
```
所有 Transform 数据连续存储：
[Transform1][Transform2][Transform3]...

CPU 缓存命中率高，性能好
```

**3. 并行处理**
```
MovementSystem 可以并行处理所有实体
不同 System 可以并行执行
```

### 6.2 ECS 在游戏中的应用

**Unity DOTS（Data-Oriented Technology Stack）：**
- Unity 的 ECS 实现
- 高性能
- 适合大规模对象

**Unreal Engine：**
- 传统 OOP + Component 模式
- 不是纯 ECS
- 但可以手动实现 ECS 思想

**游戏场景应用：**

**大量小兵：**
```
每个小兵是一个 Entity
Components: Transform, Health, AI, Team
Systems:
- AISystem: 更新所有小兵的 AI
- HealthSystem: 处理所有小兵的血量
- RenderSystem: 渲染所有小兵
```

**粒子系统：**
```
每个粒子是一个 Entity
Components: Transform, Velocity, Lifetime, Color
Systems:
- ParticleUpdateSystem: 更新所有粒子
- ParticleRenderSystem: 渲染所有粒子
```

## 第三部分：性能优化

## 七、渲染优化

### 7.1 渲染管线基础

**渲染管线流程：**
```
应用阶段（CPU）
    ↓
几何阶段（GPU）
    ↓
光栅化阶段（GPU）
```

**1. 应用阶段（CPU）**
- 场景管理
- 视锥剔除
- 遮挡剔除
- 准备渲染数据
- 提交 Draw Call

**2. 几何阶段（GPU）**
- 顶点着色器
- 曲面细分
- 几何着色器
- 裁剪
- 屏幕映射

**3. 光栅化阶段（GPU）**
- 三角形设置
- 三角形遍历
- 片元着色器
- 深度测试
- 混合

### 7.2 Draw Call 优化

**什么是 Draw Call？**

CPU 告诉 GPU "画这个物体"的指令。

**为什么 Draw Call 多会卡？**

每个 Draw Call 都有开销：
- CPU 准备数据
- CPU 和 GPU 通信
- GPU 切换状态

**优化方法：**

**1. 批处理（Batching）**

**静态批处理：**
```
场景中 100 个相同的树
不优化：100 个 Draw Call
优化后：合并成 1 个 Draw Call
```

**动态批处理：**
```
运行时合并小物体
限制：顶点数不能太多（通常 < 300）
```

**2. GPU Instancing**
```
100 个相同的敌人
不优化：100 个 Draw Call
优化后：1 个 Draw Call + 100 个实例数据
```

**实现：**
```
绘制 100 个敌人：
DrawInstanced(mesh, material, 100, instanceData)

instanceData 包含每个实例的：
- 位置
- 旋转
- 缩放
- 颜色
```

**3. 合并材质（Material Merging）**
```
10 个物体，10 个不同材质
优化：合并成 1 个材质（纹理图集）
```

**游戏场景应用：**
- 小兵：使用 GPU Instancing
- 场景物体：静态批处理
- UI 元素：动态批处理

### 7.3 LOD（Level of Detail）

**核心思想：**
根据距离使用不同精度的模型。

**LOD 级别：**
```
LOD 0（近距离）：10000 三角形
LOD 1（中距离）：5000 三角形
LOD 2（远距离）：1000 三角形
LOD 3（超远）：100 三角形
```

**切换策略：**

**硬切换：**
```
if (distance < 1000) {
    使用 LOD 0
} else if (distance < 3000) {
    使用 LOD 1
} else {
    使用 LOD 2
}
```

**问题：**切换时有明显跳变

**软切换（Dithering）：**
```
在切换距离附近，混合两个 LOD
使用抖动技术平滑过渡
```

**HLOD（Hierarchical LOD）：**
```
远处的多个建筑合并成一个低模
```

**游戏场景应用：**
- 角色：3-4 个 LOD 级别
- 场景：HLOD 合并远处建筑
- 植被：Impostor（远处用贴图替代）

### 7.4 遮挡剔除（Occlusion Culling）

**问题：**
渲染被遮挡的物体浪费性能。

**解决方案：**

**1. 视锥剔除（Frustum Culling）**
```
不在相机视野内的物体不渲染
```

**2. 遮挡剔除（Occlusion Culling）**
```
被其他物体完全遮挡的物体不渲染
```

**实现方法：**

**预计算遮挡：**
```
离线计算每个位置能看到哪些物体
运行时查表
```

**动态遮挡查询：**
```
使用 GPU 查询物体是否可见
```

**游戏场景应用：**
- 室内场景：预计算遮挡
- 开放世界：动态遮挡查询

### 7.5 纹理优化

**1. 纹理压缩**
```
未压缩：1024x1024 RGBA = 4MB
压缩后：1024x1024 DXT5 = 1MB
```

**2. Mipmap**
```
远处物体使用低分辨率纹理
减少显存带宽
避免摩尔纹
```

**3. 纹理图集（Texture Atlas）**
```
多个小纹理合并成一个大纹理
减少材质切换
```

**4. 流式加载**
```
只加载当前需要的纹理
远处物体使用低分辨率版本
```

## 八、内存管理

### 8.1 内存泄漏的常见原因

**1. 循环引用**

**问题：**
```
角色持有武器的强引用
武器持有角色的强引用
两者都无法释放
```

**解决：**
```
至少一方使用弱引用
武器持有角色的弱引用
```

**2. 未解绑的委托/事件**

**问题：**
```
对象 A 订阅了对象 B 的事件
对象 A 销毁时没有取消订阅
对象 B 仍然持有对象 A 的引用
```

**解决：**
```
在对象销毁时（EndPlay/Destructor）
取消所有订阅
```

**3. 未清理的定时器**

**问题：**
```
设置了定时器
对象销毁时定时器仍在运行
定时器持有对象引用
```

**解决：**
```
在对象销毁时清理所有定时器
```

**4. 资源未释放**

**问题：**
```
加载了资源但从未释放
资源一直占用内存
```

**解决：**
```
使用引用计数
不再使用时释放资源
```

### 8.2 对象池（Object Pool）

**问题：**
频繁创建和销毁对象导致：
- 内存分配开销
- 垃圾回收压力
- 内存碎片

**解决方案：对象池**

**核心思想：**
```
预先创建一批对象
需要时从池中取出
不需要时放回池中
```

**实现：**
```
初始化：
    创建 100 个子弹对象
    全部放入池中（未激活）

需要子弹时：
    从池中取出一个
    激活并初始化
    使用

子弹消失时：
    停用
    放回池中
```

**优点：**
- 避免频繁分配内存
- 性能稳定
- 减少 GC 压力

**适用场景：**
- 子弹
- 特效
- 敌人
- UI 元素

**游戏场景应用：**
在你的项目中：
- 小兵对象池
- 技能特效对象池
- 伤害数字对象池

### 8.3 内存分配策略

**1. 栈分配 vs 堆分配**

**栈分配：**
```
void Function() {
    int value = 10;  // 栈分配
    // 函数结束自动释放
}
```
- 速度快
- 自动管理
- 大小有限

**堆分配：**
```
void Function() {
    int* value = new int(10);  // 堆分配
    // 需要手动 delete
}
```
- 大小灵活
- 需要手动管理
- 速度较慢

**2. 内存池（Memory Pool）**

**核心思想：**
```
预先分配一大块内存
从这块内存中分配小对象
避免频繁调用系统分配函数
```

**3. 自定义分配器**

**针对特定场景优化：**
```
线性分配器：顺序分配，一次性释放
栈分配器：后进先出
池分配器：固定大小对象
```

## 九、多线程与异步

### 9.1 游戏中的多线程

**主线程（Game Thread）：**
- 游戏逻辑
- 输入处理
- 脚本执行

**渲染线程（Render Thread）：**
- 准备渲染数据
- 提交 Draw Call

**RHI 线程（GPU Thread）：**
- 与 GPU 通信
- 执行渲染命令

**工作线程（Worker Threads）：**
- 异步任务
- 资源加载
- 物理计算

### 9.2 异步加载

**问题：**
同步加载大资源会卡顿。

**解决方案：异步加载**

**实现：**
```
开始异步加载：
    创建加载任务
    在工作线程中加载
    主线程继续运行

加载完成：
    回调通知主线程
    主线程使用资源
```

**注意事项：**
- 加载完成前显示占位资源
- 显示加载进度
- 支持取消加载

**游戏场景应用：**
- 关卡流式加载
- 纹理异步加载
- 音频异步加载

### 9.3 线程安全

**问题：**
多个线程访问同一数据导致竞态条件。

**解决方案：**

**1. 互斥锁（Mutex）**
```
同一时间只有一个线程可以访问
```

**2. 读写锁（Read-Write Lock）**
```
多个线程可以同时读
只有一个线程可以写
```

**3. 原子操作（Atomic）**
```
不可分割的操作
不需要锁
```

**4. 无锁数据结构**
```
使用原子操作实现
高性能
```

**游戏中的线程安全：**
- 游戏逻辑在主线程
- 渲染数据在渲染线程
- 使用消息队列通信

## 十、技能系统设计

### 10.1 技能系统架构

**核心组件：**

**1. 技能定义（Skill Definition）**
```
技能 ID
技能名称
技能描述
冷却时间
消耗（蓝量/能量）
施法距离
作用范围
伤害/效果
动画
特效
音效
```

**2. 技能实例（Skill Instance）**
```
当前冷却剩余时间
当前等级
是否可用
```

**3. 技能管理器（Skill Manager）**
```
管理角色的所有技能
处理技能释放
处理技能升级
处理技能冷却
```

### 10.2 技能释放流程

**完整流程：**

**1. 输入检测**
```
玩家按下技能键
```

**2. 客户端验证**
```
检查：
- 技能是否存在
- CD 是否好了
- 蓝量是否足够
- 是否在施法中
- 是否被控制（眩晕等）

如果验证失败：
- 显示错误提示
- 播放错误音效
- 不发送给服务器

如果验证通过：
- 继续下一步
```

**3. 客户端预表现**
```
立即播放：
- 施法动画
- 施法特效
- 施法音效

给玩家即时反馈
```

**4. 发送给服务器**
```
ServerCastSkill(SkillID, TargetLocation)
```

**5. 服务器验证**
```
严格检查：
- 技能是否存在
- CD 是否好了（服务器时间）
- 蓝量是否足够
- 目标是否在范围内
- 是否有视野
- 是否被控制

如果验证失败：
- 通知客户端取消
- 记录可疑行为（可能作弊）

如果验证通过：
- 继续下一步
```

**6. 服务器执行**
```
扣除资源：
- 消耗蓝量
- 设置 CD

计算效果：
- 计算伤害
- 应用 Buff/Debuff
- 触发特殊效果

记录日志：
- 技能释放时间
- 施法者
- 目标
- 结果
```

**7. 服务器广播**
```
通知所有相关客户端：
- 技能释放者
- 技能目标
- 附近玩家

MulticastPlaySkillEffect(SkillID, Location, Target)
```

**8. 客户端表现**
```
其他玩家的客户端：
- 播放技能动画
- 播放技能特效
- 播放技能音效
- 显示伤害数字
```

### 10.3 技能类型设计

**1. 瞬发技能（Instant）**
```
特点：
- 立即生效
- 无施法时间

示例：
- 闪现
- 治疗术
- 净化

实现：
- 按下立即执行
- 无需等待
```

**2. 引导技能（Channeling）**
```
特点：
- 需要持续施法
- 可以被打断
- 移动会中断

示例：
- 传送
- 持续治疗
- 蓄力攻击

实现：
- 开始引导
- 每帧检查是否中断
- 完成后生效
```

**3. 吟唱技能（Casting）**
```
特点：
- 有施法前摇
- 可以被打断
- 完成后生效

示例：
- 火球术
- 大招

实现：
- 播放施法动画
- 等待施法时间
- 释放技能效果
```

**4. 投射物技能（Projectile）**
```
特点：
- 发射投射物
- 有飞行时间
- 可以被躲避

示例：
- 箭矢
- 魔法弹
- 手榴弹

实现：
- 生成投射物
- 投射物飞行
- 碰撞检测
- 命中生效
```

**5. 范围技能（AOE）**
```
特点：
- 作用于区域
- 影响多个目标

示例：
- 地震
- 火雨
- 爆炸

实现：
- 确定作用范围
- 查找范围内所有目标
- 对每个目标应用效果
```

### 10.4 技能冷却系统

**冷却类型：**

**1. 独立冷却**
```
每个技能有自己的 CD
互不影响
```

**2. 共享冷却**
```
一组技能共享 CD
释放一个，其他也进入 CD
```

**3. 充能系统**
```
技能有多个充能
每个充能独立冷却

示例：
- 闪现有 2 层充能
- 使用一层，该层开始 CD
- 另一层仍可使用
```

**实现：**
```
技能数据：
- CooldownDuration: 10 秒
- CooldownEndTime: 服务器时间戳

检查是否可用：
if (CurrentServerTime >= CooldownEndTime) {
    可以使用
} else {
    剩余时间 = CooldownEndTime - CurrentServerTime
}

释放技能：
CooldownEndTime = CurrentServerTime + CooldownDuration
```

**冷却缩减（CDR）：**
```
实际 CD = 基础 CD * (1 - CDR%)

示例：
基础 CD: 10 秒
CDR: 40%
实际 CD: 10 * (1 - 0.4) = 6 秒
```

### 10.5 Buff/Debuff 系统

**Buff 类型：**

**1. 属性修改**
```
增加攻击力
增加移动速度
增加护甲
```

**2. 状态效果**
```

**3. 持续效果**
```
持续治疗（HOT）
持续伤害（DOT）
持续回蓝
```

**Buff 数据结构：**
```
Buff ID
Buff 名称
持续时间
效果类型
效果数值
是否可叠加
最大叠加层数
图标

**Buff 管理：**

1. 检查是否已存在

5. 显示图标

## 十一、AI 系统

### 11.1 行为树（Behavior Tree）

**核心概念：**

**节点类型：**

**1. 复合节点（Composite）**

**Sequence（顺序）：**
```
从左到右执行子节点
所有成功才成功
一个失败就失败

示例：
Sequence
├─ 检查敌人在视野内
├─ 移动到攻击范围
└─ 攻击

如果任何一步失败，整个失败
```

**Selector（选择）：**
```
从左到右执行子节点
一个成功就成功
所有失败才失败

示例：
Selector
├─ 攻击敌人（如果有敌人）
├─ 巡逻（如果没敌人）
└─ 待机

优先执行第一个，失败才执行下一个
```

**Parallel（并行）：**
```
同时执行多个子节点

示例：
Parallel
├─ 移动到目标
└─ 播放移动动画
```

**2. 装饰节点（Decorator）**

**条件检查：**
```
Decorator: 血量 < 30%
└─ 逃跑

只有满足条件才执行子节点
```

**循环：**
```
Decorator: 重复 3 次
└─ 攻击

重复执行子节点
```

**3. 叶子节点（Leaf）**

**任务节点（Task）：**
```
执行具体动作：
- 移动到位置
- 攻击目标
- 播放动画
```

**条件节点（Condition）：**
```
检查条件：
- 敌人在范围内？
- 血量低于 50%？
- 有弹药？
```

### 11.2 黑板系统（Blackboard）

**核心思想：**
共享数据存储，行为树节点通过黑板通信。

**黑板数据：**
```
Enemy: 当前目标敌人
PatrolPoint: 巡逻点
IsAlert: 是否警戒
LastSeenLocation: 最后看到敌人的位置
```

**使用示例：**
```
任务：寻找敌人
成功：设置黑板 Enemy = 找到的敌人

任务：攻击敌人
读取黑板 Enemy
攻击该敌人
```

### 11.3 AI 感知系统

**感知类型：**

**1. 视觉（Sight）**
```
检测视野内的对象
考虑：
- 视野角度（FOV）
- 视野距离
- 遮挡（是否被墙挡住）
```

**2. 听觉（Hearing）**
```
检测声音事件
考虑：
- 声音距离
- 声音大小
```

**3. 伤害感知（Damage）**
```
受到伤害时感知攻击者
```

**实现：**
```
每帧更新：
1. 检测视野内的对象
2. 检测听到的声音
3. 更新黑板数据

感知到敌人：
1. 设置黑板 Enemy
2. 触发警戒状态
3. 行为树切换到战斗分支
```

### 11.4 寻路系统

**A* 算法：**

**核心思想：**
```
F = G + H

G: 从起点到当前点的实际代价
H: 从当前点到终点的估计代价（启发式）
F: 总代价

选择 F 值最小的节点扩展
```

**实现步骤：**
```
1. 将起点加入开放列表
2. 循环：
   a. 从开放列表选择 F 值最小的节点
   b. 如果是终点，路径找到
   c. 否则，扩展该节点的邻居
   d. 计算邻居的 G、H、F 值
   e. 将邻居加入开放列表
3. 如果开放列表为空，路径不存在
```

**优化：**

**1. 导航网格（NavMesh）**
```
预计算可行走区域
运行时在网格上寻路
比网格寻路快得多
```

**2. 分层寻路**
```
先在高层网格寻路（粗略路径）
再在低层网格寻路（精确路径）
```

**3. 路径平滑**
```
寻路结果是折线
平滑成曲线
更自然
```

**游戏场景应用：**
在你的项目中：
- 小兵使用 NavMesh 寻路
- 行为树控制小兵行为
- 黑板存储目标信息

## 十二、物理系统

### 12.1 碰撞检测

**碰撞检测阶段：**

**1. 粗略阶段（Broad Phase）**
```
目标：快速排除不可能碰撞的对象对
方法：空间分区

常用算法：
- 网格划分
- 四叉树/八叉树
- 扫描线算法
```

**2. 精确阶段（Narrow Phase）**
```
目标：精确检测碰撞
方法：几何算法

常用算法：
- AABB vs AABB
- 球体 vs 球体
- OBB vs OBB
- 凸包 vs 凸包（GJK 算法）
```

**碰撞形状：**

**AABB（轴对齐包围盒）：**
```
特点：
- 与坐标轴对齐
- 检测快速
- 旋转后需要重新计算

检测算法：
if (box1.max.x < box2.min.x || box1.min.x > box2.max.x) return false;
if (box1.max.y < box2.min.y || box1.min.y > box2.max.y) return false;
if (box1.max.z < box2.min.z || box1.min.z > box2.max.z) return false;
return true;
```

**球体（Sphere）：**
```
特点：
- 最简单
- 旋转不影响
- 不够精确

检测算法：
distance = Length(center1 - center2);
return distance < (radius1 + radius2);
```

**OBB（方向包围盒）：**
```
特点：
- 可以旋转
- 更贴合物体
- 检测较复杂

检测算法：
使用分离轴定理（SAT）
```

**凸包（Convex Hull）：**
```
特点：
- 最精确
- 计算最复杂

检测算法：
GJK 算法
```

### 12.2 物理引擎集成

**常见物理引擎：**
- PhysX（Unreal Engine 默认）
- Havok
- Bullet
- Box2D（2D）

**物理对象类型：**

**1. 静态对象（Static）**
```
特点：
- 不移动
- 不受力影响
- 只参与碰撞检测

示例：
- 地面
- 墙壁
- 建筑
```

**2. 动态对象（Dynamic）**
```
特点：
- 可以移动
- 受力影响
- 有质量

示例：
- 箱子
- 球
- 可破坏物体
```

**3. 运动学对象（Kinematic）**
```
特点：
- 可以移动
- 不受力影响
- 由代码控制

示例：
- 玩家角色
- 电梯
- 移动平台
```

**物理材质：**
```
摩擦力：
- 静摩擦
- 动摩擦

弹性：
- 反弹系数
- 0 = 完全不反弹
- 1 = 完全弹性碰撞

密度：
- 影响质量
```

### 12.3 角色控制器

**问题：**
直接使用物理模拟控制角色会有问题：
- 容易被推动
- 爬坡困难
- 楼梯卡顿

**解决方案：角色控制器（Character Controller）**

**特点：**
- 胶囊体碰撞
- 自动爬坡
- 自动上楼梯
- 不会被推动（或可配置）

**实现：**
```
每帧更新：
1. 获取输入方向
2. 计算移动向量
3. 检测碰撞
4. 滑动处理（沿墙移动）
5. 应用重力
6. 更新位置
```

**爬坡处理：**
```
if (坡度 < 最大爬坡角度) {
    沿坡面移动
} else {
    阻止移动
}
```

**楼梯处理：**
```
if (台阶高度 < 最大台阶高度) {
    自动抬升
} else {
    需要跳跃
}
```

## 十三、热更新

### 13.1 热更新原理

**核心思想：**
在不重新安装游戏的情况下，更新游戏内容。

**可热更新的内容：**
- 资源文件（模型、贴图、音频）
- 配置文件（数值、文本）
- 脚本代码（Lua、C#）
- 关卡数据

**不可热更新的内容：**
- 引擎代码（C++）
- 底层逻辑

### 13.2 热更新流程

**1. 版本检查**
```
客户端启动：
1. 读取本地版本号
2. 向服务器请求最新版本号
3. 对比版本号

if (本地版本 < 服务器版本) {
    需要更新
} else {
    无需更新
}
```

**2. 获取更新列表**
```
服务器返回：
{
    "version": "1.0.1",
    "files": [
        {
            "path": "Content/Maps/Level1.pak",
            "size": 1024000,
            "md5": "abc123..."
        },
        {
            "path": "Config/Items.json",
            "size": 5000,
            "md5": "def456..."
        }
    ]
}
```

**3. 下载更新文件**
```
for each file in files:
    if (本地文件不存在 || MD5 不匹配) {
        下载文件
        显示进度
    }
```

**4. 校验文件**
```
for each file:
    计算 MD5
    if (MD5 != 预期 MD5) {
        文件损坏，重新下载
    }
```

**5. 应用更新**
```
替换旧文件
更新版本号
重启游戏（如果需要）
```

### 13.3 增量更新

**问题：**
每次都下载完整文件浪费流量。

**解决方案：**
只下载变化的部分。

**差分算法：**
```
1. 计算新旧文件的差异
2. 生成差分包（Patch）
3. 客户端下载差分包
4. 应用差分包到旧文件
5. 生成新文件
```

**常用算法：**
- BSDiff（二进制差分）
- XDelta（增量编码）

**优点：**
- 节省流量
- 下载更快

**缺点：**
- 需要旧文件
- 应用差分需要时间

### 13.4 资源热重载

**问题：**
更新资源后需要重启游戏。

**解决方案：**
运行时重新加载资源。

**实现：**
```
监听文件变化：
1. 检测资源文件修改
2. 卸载旧资源
3. 重新加载新资源
4. 通知使用该资源的对象

对象更新：
1. 收到资源更新通知
2. 替换旧资源引用
3. 刷新显示
```

**注意事项：**
- 确保资源引用正确更新
- 处理资源加载失败
- 避免内存泄漏

## 十四、资源管理

### 14.1 资源加载策略

**1. 预加载（Preload）**
```
游戏启动时加载常用资源
优点：使用时无需等待
缺点：启动时间长，内存占用大
```

**2. 按需加载（On-Demand）**
```
需要时才加载
优点：启动快，内存占用小
缺点：首次使用有延迟
```

**3. 流式加载（Streaming）**
```
根据玩家位置动态加载
优点：内存占用可控
缺点：实现复杂
```

**4. 分级加载**
```
先加载低质量版本
再加载高质量版本
优点：快速显示，逐步提升
```

### 14.2 资源引用计数

**问题：**
如何知道资源是否还在使用？

**解决方案：引用计数**

**实现：**
```
加载资源：
1. 检查是否已加载
2. 如果已加载，引用计数 +1
3. 如果未加载，加载资源，引用计数 = 1
4. 返回资源

释放资源：
1. 引用计数 -1
2. 如果引用计数 = 0，卸载资源
```

**智能指针：**
```
使用智能指针自动管理引用计数
TSharedPtr（Unreal）
shared_ptr（C++）
```

### 14.3 资源打包

**PAK 文件：**
```
将多个资源文件打包成一个文件
优点：
- 减少文件数量
- 加快加载速度
- 方便分发
```

**AssetBundle（Unity）：**
```
资源包系统
支持：
- 按需下载
- 增量更新
- 资源共享
```

### 示例回答

**问题：TCP 三次握手为什么需要三次？**

**差的回答：**
"为了建立可靠连接。"

**好的回答：**
"TCP 三次握手需要三次主要有三个原因：

第一，防止失效的连接请求突然传到服务器。比如客户端发送了一个连接请求，但因为网络延迟，这个请求在网络中滞留了很久。客户端以为请求丢失，又发送了新的请求并成功建立连接。后来第一个请求终于到达服务器，如果是两次握手，服务器会建立连接，但客户端已经不需要这个连接了，造成资源浪费。三次握手可以避免这个问题。

第二，确认双方的收发能力。第一次握手确认客户端的发送能力和服务器的接收能力，第二次握手确认服务器的发送能力和客户端的接收能力，第三次握手确认客户端的接收能力。

第三，协商初始序列号。TCP 是可靠传输，需要序列号来保证数据顺序和完整性。三次握手过程中，双方交换初始序列号，为后续数据传输做准备。

在我的项目中，玩家连接游戏服务器时就是通过 TCP 三次握手建立连接的。如果握手失败，客户端会提示'连接服务器失败，请检查网络'。"
