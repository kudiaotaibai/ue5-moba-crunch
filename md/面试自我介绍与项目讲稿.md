## 三、项目讲稿一：UE5 在线 MOBA 技术原型

### 1. 一句话版本

这是一个我独立完成的 UE5 在线 MOBA 技术原型，重点是把多人对局框架、GAS 战斗系统、UI 和 Dedicated Server 联机链路完整打通。

### 2. 1 分钟讲稿

这个项目我主要想验证两件事：第一，UE5 下 MOBA 这类多人对战项目的基础架构应该怎么搭；第二，GAS、Session 和 Dedicated Server 这些模块怎么真正串起来。

我在项目里先搭了多人对局的 Gameplay Framework，包括 GameMode、GameState、PlayerController、PlayerState 和 Character 的职责划分，然后基于 GAS 做了技能、属性成长、Buff / Debuff、升级点和死亡流程。UI 这块我用 UMG 做了 HUD、商店、计分板等界面，并通过 C++ 事件和属性更新做联动。

联机这块我还自己实现了客户端建房到加入对局的链路：客户端请求协调器创建对局，协调器分配端口并拉起服务器实例，Dedicated Server 创建 Session，客户端再按搜索结果加入并 Travel。

这个项目对我最大的价值是让我把 UE 客户端开发里最核心的几条链都实际走通了，而不是只停留在单机玩法层面。

### 3. 2 到 3 分钟讲稿

这个项目是我独立完成的一个 UE5 在线 MOBA 技术原型，核心目标不是把它做成完整商业游戏，而是把 MOBA 类型里最关键的客户端和联机链路真正做通。

我一开始先搭的是多人对局框架。可以把这一层理解成一套“角色、规则、状态和输入”的分工系统：GameMode 负责规则，GameState 负责全局状态，PlayerController 负责输入和 UI 交互，PlayerState 负责玩家状态数据，Character 负责具体表现和角色能力。这个分层一旦清楚，后面的功能就比较容易往上挂。

战斗系统我用的是 GAS。因为如果技能、属性、Buff、Debuff 全都手写散在各个类里，后期会像把很多电线直接缠在一起，很难扩展。GAS 更像一个统一插线板，我在项目里基于它实现了属性成长、技能释放、控制效果、伤害结算和升级流程，也做了多段连击、AOE 和投射物这些能力。

UI 这块我主要做了 HUD、商店和计分板，并用 C++ 去监听属性变化和事件回调，避免 UI 和角色逻辑直接写死耦合。

联机部分是这个项目我投入比较多的一块。我没有只停留在本地 Listen Server，而是自己把 Session 和 Dedicated Server 链路做了一套可跑通的原型。大致流程是：客户端先请求协调器创建对局，协调器用 Flask 处理建房请求并分配端口，再拉起服务器实例；Dedicated Server 启动后创建 Session，客户端再进行查找、加入和地图切换。这个过程也让我对服务端权威、Session 生命周期和客户端加入流程有了更实际的理解。

如果说这个项目里我最在意的收获，我会说有两个。第一是我对 UE 客户端整体架构的感觉更完整了，不再只是会单点功能。第二是我真正踩过 GAS、联机和服务器启动链路这些工程问题，而不是只看过教程。

### 4. 常见追问与回答点

#### Q1：为什么要用 GAS？

因为技能、属性、Buff 和状态管理如果完全手写，后期会越来越乱。GAS 的价值在于把这些逻辑抽象成统一的数据和执行框架，更适合复杂战斗系统扩展。

#### Q2：GameMode 和 GameState 的区别是什么？

GameMode 更像服务器上的规则管理器，决定对局规则和流程；GameState 更像同步给所有玩家的全局状态板，负责让客户端看到一致的对局状态。

#### Q3：你这个项目最难的点是什么？

我觉得最难的是把客户端框架、GAS 和联机链路真正串成一个可运行系统。因为单看每块都能理解，但一旦它们放到多人对局里，状态同步、职责划分和生命周期问题会一下子变复杂。

#### Q4：如果继续做下去你会优先补什么？

我会优先补更稳定的对局流程和更完整的玩法循环，比如断线重连、匹配流程和更细的战斗数值验证。

## 四、项目讲稿二：UE5 RDG 渲染插件开发

### 1. 一句话版本

这是一个我独立完成的 UE5 RDG 渲染插件项目，目标是从最小后处理 pass 出发，逐步实现屏幕空间调试视图、Blur、SSAO 和 AO Composite 的完整链路。

### 2. 1 分钟讲稿

这个项目我主要是想补图形学和 UE 渲染底层能力，所以没有直接上大而全的渲染器，而是选了 UE5 的 RDG 渲染插件路线。

我先做的是最小可运行全屏 pass，然后一步步实现了 Depth、Normal、Roughness、SceneColor 这些调试视图。接着我实现了 Gaussian Blur 和 Bilateral Blur，再往上做了 Raw SSAO、降噪后的 SSAO 和 AO Composite。

这个项目里我真正做的是一条完整的屏幕空间后处理链：从 SceneDepth 和 GBuffer 里取数据，重建位置和法线，做半球采样和遮挡估计，得到原始 AO，再通过双边模糊降噪，最后合成回画面。

为了方便调试，我还做了一个 Slate 调参窗口，可以实时切换模式和修改 Blur、SSAO、Composite 参数。

### 3. 2 到 3 分钟讲稿

这个项目本质上是我自己做的一套 UE5 屏幕空间渲染调试和 AO 后处理插件。之所以做它，是因为我之前更多做的是 gameplay 和联机，我想补一块更底层的图形学和渲染能力。

我没有一上来就做很大的东西，而是先从最小的全屏渲染 pass 开始。先把 Project Plugin、Scene View Extension、Global Shader 和 RDG 这条接入链路打通。可以把它理解成我先搭了一个“后处理实验台”，只有实验台搭好，后面每一个效果才能稳定往上叠。

第一阶段我做的是调试视图，比如 Depth、Normal、Roughness 和 SceneColor。因为如果连输入数据长什么样都看不到，后面做任何屏幕空间效果都会像闭眼调程序。

第二阶段我做了 Gaussian Blur 和 Bilateral Blur。Gaussian Blur 更像简单平均邻域像素，Bilateral Blur 会额外参考深度和法线信息，尽量只在同一表面内部平滑，避免把边界糊穿。

第三阶段我做的是 SSAO。它的核心思路可以理解成：站在当前像素这个点上，像往周围打出一圈小探针，看看周围方向是不是很快就撞到别的几何体。如果很多方向都被挡住，就说明这个点环境光比较难进来，AO 就更黑。实现上我先用深度重建出当前位置，再结合 GBuffer 法线建立局部半球采样方向，做遮挡判断，得到 Raw SSAO，然后再接 normal-aware bilateral blur 去降噪。

最后我又做了 AO Composite，把模糊后的 AO 结果作用回 SceneColor，并额外给合成阶段做了 Strength、Power 和 MinVisibility 这些参数，让效果不只是“直接乘黑”。

这个项目让我收获最大的地方有两个。第一，我真正把 RDG 的 pass 组织和资源依赖走通了，不只是会调用 API。第二，我对 GBuffer、深度重建、屏幕空间采样和后处理链路的理解从“知道名词”变成了“我自己做过一遍”。

### 4. 常见追问与回答点

#### Q1：RDG 是什么？为什么用它？

RDG 可以理解成 UE 渲染里用来组织 pass 和资源依赖关系的图。对我这个项目来说，它的价值是让我能清楚表达每一步输入输出，比如 RawSSAO 先写到中间纹理，再交给双边模糊 pass，最后再交给 AOComposite。

#### Q2：为什么先做 Depth / Normal / Roughness 调试视图？

因为这些就是后面很多屏幕空间效果的输入。先把输入看清楚，相当于先把调试仪表盘搭起来，不然后面 AO 出问题时很难知道是深度错了、法线错了还是采样逻辑错了。

#### Q3：Raw SSAO 为什么会有噪点？

因为它本质上是有限样本的近似估计。采样数不高、采样核有随机旋转，再加上遮挡判断本身比较硬，所以会有颗粒感。

#### Q4：为什么 Bilateral Blur 比 Gaussian Blur 更适合 SSAO？

因为 Gaussian Blur 只看距离，会把边界两边的值一起平均掉；Bilateral Blur 会看深度和法线差异，更像“只模糊属于同一表面的一群像素”，所以更适合给 SSAO 这种边界敏感的结果降噪。

#### Q5：AO Composite 是不是物理正确的？

我这版不是物理正确的 lighting integration，而是后处理式 AO 合成。也就是先得到 AO，再用一条可调曲线把它乘回 SceneColor。它更适合学习和验证整条链路，但我也知道和引擎真正 lighting 阶段集成的 AO 不是一回事。

#### Q6：这个项目里最难的点是什么？

我觉得最难的不是某一个公式，而是整条链的打通和排错。比如扩展注册时机、shader 参数绑定、全局 shader 编译错误、pass 之间资源传递和模式切换，这些都需要一点点收口。
