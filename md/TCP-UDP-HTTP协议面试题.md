## TCP传输控制协议

### 基本概念
TCP (Transmission Control Protocol) 是一种面向连接的、可靠的、基于字节流的传输层通信协议。

### 核心特性
- **面向连接**：通信前必须建立连接
- **可靠传输**：保证数据完整、有序到达
- **流量控制**：防止发送方发送过快
- **拥塞控制**：防止网络过载
- **全双工通信**：双向同时传输数据

### TCP三次握手（建立连接）
```
客户端                    服务器
  |                         |
  |----SYN(seq=x)---------->|  第一次握手：客户端发送SYN
  |                         |
  |<---SYN+ACK(seq=y,ack=x+1)|  第二次握手：服务器回应SYN+ACK
  |                         |
  |----ACK(ack=y+1)-------->|  第三次握手：客户端确认ACK
  |                         |
  |    连接建立完成          |
```

**为什么需要三次握手？**
- 防止已失效的连接请求突然传到服务器，造成资源浪费
- 确认双方的收发能力都正常
- 协商初始序列号，保证数据传输的可靠性

### TCP四次挥手（断开连接）
```
客户端                    服务器
  |                         |
  |----FIN(seq=u)---------->|  第一次挥手：客户端请求关闭
  |                         |
  |<---ACK(ack=u+1)---------|  第二次挥手：服务器确认
  |                         |
  |<---FIN(seq=w)-----------|  第三次挥手：服务器请求关闭
  |                         |
  |----ACK(ack=w+1)-------->|  第四次挥手：客户端确认
  |                         |
  |    连接关闭完成          |
```

**为什么需要四次挥手？**
- TCP是全双工通信，双方都需要主动关闭
- 服务器收到FIN后可能还有数据要发送
- 确保双方数据都传输完毕

### TCP可靠性保证机制

#### 1. 序列号与确认应答
- 每个字节都有序列号
- 接收方返回ACK确认收到的数据

#### 2. 超时重传
- 发送数据后启动定时器
- 超时未收到ACK则重传

#### 3. 流量控制（滑动窗口）
- 接收方通告窗口大小
- 发送方根据窗口大小控制发送速率

#### 4. 拥塞控制
- **慢启动**：指数增长发送窗口
- **拥塞避免**：线性增长
- **快速重传**：收到3个重复ACK立即重传
- **快速恢复**：减半窗口后继续传输

### TCP报文格式
```
0                   16                  31
+-------------------+-------------------+
|    源端口号        |    目的端口号      |
+-------------------+-------------------+
|            序列号 (Sequence Number)    |
+---------------------------------------+
|          确认号 (Acknowledgment)       |
+---------------------------------------+
|首部长度|保留|标志位|    窗口大小        |
+---------------------------------------+
|    校验和          |    紧急指针        |
+---------------------------------------+
|            选项 (可选)                 |
+---------------------------------------+
|            数据                        |
```

**重要标志位：**
- SYN：建立连接
- ACK：确认
- FIN：关闭连接
- RST：重置连接
- PSH：推送数据
- URG：紧急数据

## UDP用户数据报协议

### 基本概念
UDP (User Datagram Protocol) 是一种无连接的、不可靠的、基于数据报的传输层协议。

### 核心特性
- **无连接**：不需要建立连接，直接发送
- **不可靠**：不保证数据到达、顺序、完整性
- **面向报文**：对应用层数据不拆分不合并
- **无拥塞控制**：发送速率不受网络状态影响
- **支持一对一、一对多、多对多通信**
- **首部开销小**：仅8字节

### UDP报文格式
```
0                   16                  31
+-------------------+-------------------+
|    源端口号        |    目的端口号      |
+-------------------+-------------------+
|    UDP长度         |    校验和          |
+-------------------+-------------------+
|            数据                        |
```

### UDP的优势
1. **速度快**：无需建立连接，无需等待确认
2. **实时性好**：适合对延迟敏感的应用
3. **开销小**：首部仅8字节，TCP为20字节
4. **支持广播和组播**

### UDP的应用场景
- **实时音视频**：直播、视频会议（允许少量丢包）
- **在线游戏**：需要低延迟的游戏状态同步
- **DNS查询**：简单的请求-响应模式
- **DHCP**：动态主机配置
- **流媒体**：IPTV、网络电视
- **物联网**：传感器数据上报

## HTTP超文本传输协议

### 基本概念
HTTP (HyperText Transfer Protocol) 是应用层协议，用于在Web浏览器和服务器之间传输超文本数据。

### HTTP版本演进

#### HTTP/1.0
- 每个请求都需要建立新的TCP连接
- 无状态协议
- 支持GET、POST、HEAD方法

#### HTTP/1.1（最常用）
- **持久连接**：Connection: keep-alive
- **管道化**：可以同时发送多个请求
- **Host头**：支持虚拟主机
- **缓存控制**：Cache-Control
- **断点续传**：Range头
- 新增PUT、DELETE、OPTIONS等方法

#### HTTP/2
- **二进制分帧**：更高效的解析
- **多路复用**：一个连接并发处理多个请求
- **头部压缩**：HPACK算法
- **服务器推送**：主动推送资源

#### HTTP/3
- 基于QUIC协议（UDP）
- 更快的连接建立
- 更好的拥塞控制
- 解决队头阻塞问题

### HTTP请求方法
- **GET**：获取资源（幂等、可缓存）
- **POST**：提交数据（非幂等）
- **PUT**：更新资源（幂等）
- **DELETE**：删除资源（幂等）
- **HEAD**：获取响应头（不返回body）
- **OPTIONS**：查询支持的方法
- **PATCH**：部分更新资源
- **CONNECT**：建立隧道（用于代理）
- **TRACE**：回显请求（调试用）

### HTTP状态码

#### 1xx 信息性状态码
- 100 Continue：继续请求
- 101 Switching Protocols：切换协议

#### 2xx 成功状态码
- 200 OK：请求成功
- 201 Created：资源已创建
- 204 No Content：成功但无返回内容
- 206 Partial Content：部分内容（断点续传）

#### 3xx 重定向状态码
- 301 Moved Permanently：永久重定向
- 302 Found：临时重定向
- 304 Not Modified：资源未修改（缓存有效）
- 307 Temporary Redirect：临时重定向（保持方法）
- 308 Permanent Redirect：永久重定向（保持方法）

#### 4xx 客户端错误
- 400 Bad Request：请求语法错误
- 401 Unauthorized：未授权
- 403 Forbidden：禁止访问
- 404 Not Found：资源不存在
- 405 Method Not Allowed：方法不允许
- 408 Request Timeout：请求超时
- 429 Too Many Requests：请求过多

#### 5xx 服务器错误
- 500 Internal Server Error：服务器内部错误
- 502 Bad Gateway：网关错误
- 503 Service Unavailable：服务不可用
- 504 Gateway Timeout：网关超时

### HTTP请求报文格式
```
GET /index.html HTTP/1.1
Host: www.example.com
User-Agent: Mozilla/5.0
Accept: text/html
Connection: keep-alive
Cookie: session=abc123

[请求体，GET通常为空]
```

### HTTP响应报文格式
```
HTTP/1.1 200 OK
Content-Type: text/html; charset=utf-8
Content-Length: 1234
Cache-Control: max-age=3600
Set-Cookie: session=abc123

<!DOCTYPE html>
<html>
...
</html>
```

### HTTPS（HTTP Secure）
- HTTP + SSL/TLS加密
- 默认端口443
- 保证数据的机密性、完整性、身份认证

**HTTPS握手过程：**
1. 客户端发送支持的加密算法
2. 服务器选择算法并发送证书
3. 客户端验证证书
4. 协商会话密钥
5. 使用对称加密通信

## 协议对比与选择

### TCP vs UDP

| 特性 | TCP | UDP |
|------|-----|-----|
| 连接性 | 面向连接 | 无连接 |
| 可靠性 | 可靠传输 | 不可靠 |
| 速度 | 较慢 | 快速 |
| 首部开销 | 20字节 | 8字节 |
| 流量控制 | 有 | 无 |
| 拥塞控制 | 有 | 无 |
| 应用场景 | 文件传输、邮件、网页 | 视频、游戏、DNS |

### HTTP vs HTTPS

| 特性 | HTTP | HTTPS |
|------|------|-------|
| 安全性 | 明文传输 | 加密传输 |
| 端口 | 80 | 443 |
| 性能 | 快 | 稍慢（加密开销） |
| SEO | 一般 | 更好 |
| 证书 | 不需要 | 需要SSL证书 |

### 协议选择建议

**选择TCP的场景：**
- 需要可靠传输（文件下载、邮件）
- 数据完整性要求高
- 对顺序有要求

**选择UDP的场景：**
- 实时性要求高（游戏、直播）
- 可以容忍少量丢包
- 需要广播或组播

**选择HTTP/HTTPS的场景：**
- Web应用
- RESTful API
- 需要安全传输时用HTTPS

## 常见面试题

### 1. TCP和UDP的区别？
**答：**
- TCP面向连接，UDP无连接
- TCP可靠，UDP不可靠
- TCP有流量控制和拥塞控制，UDP没有
- TCP首部20字节，UDP首部8字节
- TCP适合文件传输，UDP适合实时通信

### 2. 为什么TCP握手是三次，挥手是四次？
**答：**
- 三次握手：确认双方收发能力，防止失效请求，协商序列号
- 四次挥手：TCP全双工，双方都要主动关闭；服务器收到FIN后可能还有数据要发，所以ACK和FIN分开发送

### 3. TCP如何保证可靠性？
**答：**
- 序列号和确认应答
- 超时重传机制
- 流量控制（滑动窗口）
- 拥塞控制（慢启动、拥塞避免、快速重传、快速恢复）
- 校验和验证数据完整性

### 4. TIME_WAIT状态是什么？为什么需要？
**答：**
- 主动关闭方在发送最后一个ACK后进入TIME_WAIT状态
- 等待时间为2MSL（Maximum Segment Lifetime）
- 目的：
  1. 确保最后的ACK能到达对方
  2. 让旧连接的数据包在网络中消失

### 5. HTTP的GET和POST有什么区别？
**答：**
- GET用于获取资源，POST用于提交数据
- GET参数在URL中，POST在请求体中
- GET有长度限制（浏览器限制），POST理论上无限制
- GET可缓存，POST不可缓存
- GET幂等，POST非幂等
- GET相对不安全（参数暴露在URL）

### 6. HTTP状态码301和302的区别？
**答：**
- 301是永久重定向，浏览器会缓存
- 302是临时重定向，不会缓存
- 301对SEO友好，权重会转移
- 302适合临时跳转场景

### 7. HTTP/1.1和HTTP/2的区别？
**答：**
- HTTP/2使用二进制分帧，HTTP/1.1是文本
- HTTP/2支持多路复用，解决队头阻塞
- HTTP/2有头部压缩（HPACK）
- HTTP/2支持服务器推送
- HTTP/2性能更好，延迟更低

### 8. HTTPS的工作原理？
**答：**
1. 客户端发起HTTPS请求
2. 服务器返回SSL证书（包含公钥）
3. 客户端验证证书有效性
4. 客户端生成随机密钥，用公钥加密发送
5. 服务器用私钥解密获得密钥
6. 双方使用该密钥进行对称加密通信

### 9. 什么是粘包和拆包？如何解决？
**答：**
- 粘包：多个小数据包合并成一个
- 拆包：一个大数据包被拆分成多个
- 原因：TCP是流式协议，没有消息边界
- 解决方案：
  1. 固定长度
  2. 特殊分隔符
  3. 消息头+长度字段
  4. 使用应用层协议（如HTTP）

### 10. 如何优化HTTP性能？
**答：**
- 使用HTTP/2或HTTP/3
- 启用GZIP压缩
- 使用CDN加速
- 合理使用缓存（Cache-Control、ETag）
- 减少请求次数（合并资源）
- 使用持久连接（keep-alive）
- 启用浏览器缓存
- 图片懒加载
- 使用WebP等高效格式

### 11. UDP如何实现可靠传输？
**答：**
- 应用层实现确认和重传机制
- 添加序列号
- 设置超时重传
- 实现流量控制
- 例如：QUIC协议就是在UDP基础上实现了可靠传输

### 12. 什么是长连接和短连接？
**答：**
- 短连接：每次通信完成后立即断开（HTTP/1.0）
- 长连接：保持连接，复用TCP连接（HTTP/1.1 keep-alive）
- 长连接优点：减少握手开销，提高性能
- 长连接缺点：占用服务器资源

### 13. Cookie和Session的区别？
**答：**
- Cookie存储在客户端，Session存储在服务器
- Cookie有大小限制（4KB），Session无限制
- Cookie相对不安全，Session更安全
- Cookie可以设置过期时间
- Session依赖Cookie传递SessionID

### 14. 什么是跨域？如何解决？
**答：**
- 跨域：浏览器同源策略限制，不同域名/端口/协议无法访问
- 解决方案：
  1. CORS（跨域资源共享）
  2. JSONP（仅支持GET）
  3. 代理服务器
  4. WebSocket
  5. postMessage

### 15. TCP的滑动窗口机制？
**答：**
- 发送窗口：允许发送但未确认的数据量
- 接收窗口：接收方能接收的数据量
- 动态调整：根据网络状况和接收方能力
- 作用：流量控制，提高传输效率

## 实际应用场景

### 游戏开发中的协议选择
- **状态同步**：UDP（位置、动作等实时数据）
- **登录认证**：TCP/HTTPS（安全性要求高）
- **聊天系统**：TCP（保证消息完整性）
- **语音通话**：UDP（实时性优先）

### Web开发中的协议应用
- **API接口**：HTTP/HTTPS + RESTful
- **实时通信**：WebSocket（基于TCP）
- **文件上传**：HTTP POST + multipart/form-data
- **视频流**：HLS（HTTP）或RTMP（TCP）

### 物联网中的协议选择
- **传感器数据**：UDP/MQTT（轻量级）
- **设备控制**：TCP/MQTT（可靠性）
- **固件升级**：TCP/HTTP（完整性）
