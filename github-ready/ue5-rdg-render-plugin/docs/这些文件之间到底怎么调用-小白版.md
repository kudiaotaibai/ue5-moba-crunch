# 这些文件之间到底怎么调用 - 小白版

## 1. 这篇文档解决什么问题

前一篇你已经知道，第一个最小版本通常会有这些文件：

- `.uplugin`
- `Build.cs`
- `Module.h / Module.cpp`
- `SceneViewExtension.h / .cpp`
- `Shaders/Private/*.usf`

但你现在真正容易卡住的地方通常不是“文件名”，而是：

- 谁先运行？
- 谁创建谁？
- 谁调用谁？
- 为什么这些文件不是直接一条函数链写到底？

这篇文档就是专门把这个调用关系讲清楚。

## 2. 先给你最终答案

最短版调用链是：

1. UE 读取 `.uplugin`
2. UE 根据 `.uplugin` 加载你的模块
3. 模块通过 `StartupModule()` 启动
4. 模块在启动时注册 Shader 目录，并创建 `Scene View Extension`
5. UE 在每帧渲染时调用这个 `Scene View Extension`
6. `Scene View Extension` 在回调里添加 RDG Pass
7. RDG Pass 使用你的 Global Shader
8. Shader 从 `.usf` 文件编译而来，并在 GPU 上执行

这就是整条链。

## 3. 一个非常重要的认知

这条链里，不是所有东西都在“直接调用下一个函数”。

有两种关系混在一起：

### 3.1 直接调用

例如：

- 模块代码里主动创建一个 `Scene View Extension` 对象
- 扩展代码里主动添加一个 RDG Pass

### 3.2 注册后由 UE 回调

例如：

- 你注册了模块，UE 之后来加载它
- 你注册了 `Scene View Extension`，UE 之后每帧渲染时来调它
- 你声明了 Shader，UE 在需要时编译并调用它

所以你要明白：

**这不是一条“你自己手写 main 函数一路 call 到底”的链，而是一条“你把几个节点挂进 UE，之后由 UE 驱动执行”的链。**

这个认知非常重要。

## 4. 第一个节点：`.uplugin`

### 4.1 它本身不执行逻辑

`.uplugin` 不是代码文件，它不会像 cpp 一样执行函数。

它更像一个配置入口。

它的作用是告诉 UE：

- 这个插件存在
- 插件里有哪些模块
- 模块该怎么加载

### 4.2 它和调用链的关系

`.uplugin` 不直接“调用”你的模块，  
但它会让 UE 知道后面该去加载哪个模块。

所以这一层的关系应该理解成：

**`.uplugin` 不是执行者，它是 UE 找到后续执行对象的线索。**

### 4.3 小白版翻译

你可以把 `.uplugin` 理解成：

**“喂，UE，我这里有个插件，你后面记得去把这个模块叫起来。”**

## 5. 第二个节点：`Build.cs`

### 5.1 它也不是运行时主角

`Build.cs` 主要发生在编译阶段，不是每帧运行时都在执行。

它告诉编译系统：

- 这个模块依赖什么
- 需要哪些引擎模块参与编译

### 5.2 它和调用链的关系

它不参与运行时调用顺序，  
但如果它配置错了：

- 模块编不过
- Renderer / RenderCore / RHI 用不了

所以它更像：

**“让整条链有资格存在的前置条件。”**

### 5.3 小白版翻译

`.uplugin` 负责“让 UE 找到你”，  
`Build.cs` 负责“让你的代码能被编出来”。

## 6. 第三个节点：`Module.cpp`

这是真正进入运行时主线的第一个代码文件。

## 7. UE 是怎么找到模块的

通常模块里会有类似这种东西：

```cpp
IMPLEMENT_MODULE(FRDGStarterModule, RDGStarter)
```

你现在不用背宏细节，只要知道它表达的是：

**“这个类就是这个模块的实现。”**

于是 UE 在加载模块时，就知道该把这个模块类拉起来。

## 8. `StartupModule()` 为什么这么重要

当 UE 真正加载你的模块后，会调用模块的启动函数。

这时模块终于开始真正干活。

在我们这个项目里，`StartupModule()` 通常负责三件事：

1. 打日志，证明模块启动了
2. 注册 Shader 目录映射
3. 创建并保存 `Scene View Extension`

所以这里是整条链上的第一个真正执行点。

## 9. 模块先做的第一件大事：注册 Shader 目录

### 9.1 为什么要注册

你的 `.usf` 文件虽然放在插件目录里，  
但 UE 并不会天然知道：

- 这些 Shader 文件在哪
- 应该怎么通过虚拟路径找到它们

所以模块启动时，通常会告诉 UE：

**“我这个插件的 Shader 目录在这里，请把它映射进来。”**

### 9.2 这一步和 `.usf` 的关系

这一步不是执行 Shader，  
而是让后面的 Shader 类有机会正确引用 `.usf` 文件。

你可以理解为：

**先把地图册交给 UE，后面 UE 才找得到你的 Shader 文件。**

## 10. 模块做的第二件大事：创建 `Scene View Extension`

### 10.1 为什么由模块来创建

因为模块是插件运行时的总控入口。

它最适合负责全局初始化对象，  
而 `Scene View Extension` 又正好是一个需要全局挂进渲染系统的对象。

所以通常关系是：

**Module 创建 Scene View Extension。**

### 10.2 创建之后还要保存

这里只讲概念，不讲具体智能指针类型。

你只要知道：

- 模块不能只临时 new 一下然后不管
- 它必须把这个扩展对象保存住

为什么？

因为以后每帧渲染时，UE 还要继续访问它。

所以这一步可以理解成：

**模块把一个“长期值班的渲染入口对象”创建出来，并留在内存里。**

## 11. `Scene View Extension` 是怎么被调用的

这一步很多小白最容易误解。

### 11.1 不是你手动每帧调用它

不是说你在 Tick 里自己写：

- `MyViewExtension->DoSomething();`

不是这种模式。

### 11.2 是 UE 在渲染流程里回调它

正确理解是：

- 模块把扩展对象挂进 UE
- UE 每次渲染视图时，在特定阶段会回调这个扩展对象

所以这个关系应该理解成：

**Module 负责注册扩展，UE 负责在每帧调用扩展。**

### 11.3 小白版翻译

模块像是给 UE 留了个电话号，  
UE 每帧渲染到某个阶段时，会主动打电话给这个扩展对象。

## 12. 扩展对象被调用后做什么

这就是你以后最核心的逻辑区。

`Scene View Extension` 被回调后，通常会做这些事：

1. 拿到当前 View 相关信息
2. 拿到当前帧的 RDG Builder 或渲染上下文
3. 检查当前是否应该启用这个效果
4. 创建中间资源
5. 准备参数
6. 添加 RDG Pass

所以 `Scene View Extension.cpp` 的核心职责不是“算图形学公式”，而是：

**在合适的时机，把你的渲染任务挂进这一帧。**

## 13. 谁创建 RDG Pass

这一步的答案是：

**通常是 `Scene View Extension` 代码在回调里创建或添加 RDG Pass。**

也就是说，关系是：

- Module 创建 View Extension
- UE 调用 View Extension
- View Extension 添加 RDG Pass

所以 RDG Pass 不是模块直接加的，  
而是通过扩展对象在“每帧的正确时机”里加进去的。

## 14. RDG Pass 和 Shader 是什么关系

这个关系要讲清楚。

### 14.1 RDG Pass 不是 Shader

Pass 是一个渲染步骤的组织单位。

Shader 是这个步骤里真正执行计算的 GPU 程序。

### 14.2 谁使用谁

更准确地说是：

**RDG Pass 会绑定并使用某个 Shader。**

例如：

- 这个 Pass 用来做深度可视化
- 那么它内部就会用一个对应的全屏 Global Shader

所以关系是：

- View Extension 添加 Pass
- Pass 使用 Shader

## 15. `.usf` 文件什么时候被“用到”

### 15.1 不是模块启动时直接执行

模块启动时只是注册 Shader 目录。

`.usf` 文件不会在那一刻直接“跑起来”。

### 15.2 真正被用到的时机

当你的 Shader 类在 C++ 侧被声明、被引用、被用于某个 Pass 时，  
UE 才会把它和 `.usf` 文件关联起来，进行编译并在 GPU 上执行。

所以 `.usf` 的角色是：

**Shader 的源码所在地。**

而不是“某个启动脚本”。

## 16. C++ 里的 Shader 类和 `.usf` 是什么关系

这里你先建立一个概念，不用深挖实现。

通常会有这样一层关系：

- `.usf` 是 GPU 代码文件
- C++ 里会有一个 Shader 类去描述它
- RDG Pass 使用这个 Shader 类

所以你可以把它理解成：

**C++ 不是直接操作一段文本 `.usf`，而是通过一个 Shader 类把 `.usf` 接进渲染流程。**

## 17. 把整个“谁创建谁，谁调用谁”重新串一遍

下面这条链建议你反复看几遍。

### 17.1 创建关系

- `.uplugin` 让 UE 知道这个插件和模块存在
- UE 加载模块
- Module 在启动时创建 `Scene View Extension`
- `Scene View Extension` 在每帧回调里添加 RDG Pass

### 17.2 调用关系

- UE 调用 `StartupModule()`
- UE 在每帧渲染时调用 `Scene View Extension`
- `Scene View Extension` 调用添加 Pass 的逻辑
- Pass 使用 Shader
- Shader 在 GPU 上运行

### 17.3 文件对应关系

- `.uplugin`
  负责“被 UE 发现”

- `Build.cs`
  负责“能编过”

- `Module.cpp`
  负责“启动并注册”

- `SceneViewExtension.cpp`
  负责“每帧接入并加 Pass”

- `.usf`
  负责“真正计算效果”

## 18. 一个最常见的误区

很多人会误以为这条链是：

- Module 直接调用 Shader

这其实不对。

更接近真实情况的是：

- Module 负责搭台子
- View Extension 负责在每帧上台
- RDG 负责安排节目顺序
- Shader 负责真正表演

所以不要把 Module 想成“直接画图的人”。

## 19. 为什么不把所有逻辑全写在 Module.cpp 里

因为那样虽然理论上能堆很多代码进去，  
但工程结构会立刻变乱。

更关键的是，职责不对。

Module 的职责是：

- 插件启动
- 注册全局对象
- 管生命周期

而不是：

- 每帧渲染细节全塞进去
- Shader 参数全塞进去
- 所有效果逻辑全塞进去

所以把渲染入口逻辑放到 `Scene View Extension` 是更自然的分层。

## 20. 为什么不直接在 Actor / GameMode 里搞这套东西

因为 Actor / GameMode 属于 Gameplay 世界。

而我们现在做的是：

**渲染流程扩展**

这两者不是一个层级的事。

你当然可以用 Actor 去控制“开关参数”，  
但真正进入每帧渲染链，还是得靠渲染入口，比如 `Scene View Extension`。

## 21. 你现在该形成的整体画面感

你以后脑子里最好能自动出现这张图：

```text
.uplugin
  -> UE 知道有这个插件
  -> 加载模块
  -> 调用 StartupModule()
  -> Module 注册 Shader 目录，创建 Scene View Extension
  -> UE 每帧渲染时回调 Scene View Extension
  -> Scene View Extension 添加 RDG Pass
  -> RDG Pass 使用 Global Shader
  -> Shader 读取 .usf 并在 GPU 上执行
```

这张图比背函数名更重要。

## 22. 如果你现在还是有点晕，就只记这 5 句话

1. `.uplugin` 让 UE 找到插件和模块。
2. Module 是插件启动后真正开始执行的地方。
3. Module 会把 `Scene View Extension` 注册进渲染系统。
4. `Scene View Extension` 会在每帧渲染时被 UE 回调，并在里面添加 RDG Pass。
5. RDG Pass 最终会调用 Global Shader，Shader 在 GPU 上运行。

如果这 5 句话你能稳定复述出来，说明整体链路已经开始成形了。

## 23. 这篇看完后你至少应该能回答的问题

- 为什么 `.uplugin` 不是“运行时代码”？
- 为什么 `Build.cs` 不是“每帧执行逻辑”？
- 为什么真正启动入口是 `StartupModule()`？
- 为什么 `Scene View Extension` 是由 Module 创建，而不是 UE 自动凭空生成？
- 为什么是 `Scene View Extension` 来加 RDG Pass，而不是 `.usf` 自己跑起来？
- 为什么 Shader 是最终计算者，而不是 Module？

## 24. 最后一句话总结

这些文件之间最核心的调用关系就是：

**UE 先通过 `.uplugin` 找到并加载模块，模块在启动时注册 Shader 并创建 `Scene View Extension`，UE 每帧渲染时回调这个扩展对象，扩展对象再用 RDG 添加 Pass，Pass 最终调用 `.usf` 对应的 Global Shader 在 GPU 上执行效果。**
